# 런칭 후 운영 프로세스

## 목적

서비스 배포 후 안정화 구간에서의 운영, 모니터링, 피드백 수집, 긴급 대응 프로세스를 정의한다.
신규 서비스의 경우 초기 안정화가 특히 중요하다.

---

## 런칭 직후 안정화 (D+0 ~ D+7)

### 모니터링 온보딩

배포 완료 시 Release AI가 다음을 자동 설정한다:

- 핵심 API 응답 시간 / 에러율 대시보드
- 서버 리소스 (CPU, 메모리, 디스크) 모니터링
- 앱 크래시 리포트 수집 (Crashlytics 등)
- 웹 Core Web Vitals 추적
- 알림 임계값 설정 (에러율 > 1%, 응답시간 > 2초 등)

### 긴급 대응 (Hotfix)

프로덕션에서 치명적 이슈 발견 시:

```
모니터링 알림 / 사용자 리포트
    ↓
AI가 로그 분석 → 원인 파악
    ↓
핫픽스 코드 생성 (Dev AI)
    ↓
긴급 리뷰 (AI + 사람 필수)
    ↓
핫픽스 배포 (사람 승인)
    ↓
배포 후 모니터링 (30분간 집중 관찰)
```

- 핫픽스는 일반 프로세스를 단축하여 진행한다
- 단, 가드레일 영역은 핫픽스에서도 사람 승인 필수이다

### 고객 커뮤니케이션

장애 또는 긴급 롤백 발생 시:

- AI가 장애 요약 + 영향 범위 + 예상 복구 시간을 정리한다
- 사람이 확인 후 고객 공지를 발행한다
- 복구 완료 후 AI가 사후 보고서 (Post-mortem) 초안을 작성한다

## 사용자 피드백 수집 (D+7~)

### 피드백 채널

- 앱 스토어 리뷰
- CS 문의 데이터
- 인앱 피드백
- 웹 사용자 행동 로그

### AI 피드백 분석

AI가 수집된 피드백을 분석하여:

- 주요 불만/요청 카테고리 분류
- 빈도 기반 우선순위 정렬
- 기존 Jira 티켓과의 중복 확인
- 새로운 Jira 티켓 초안 자동 생성 (05_ai_planning_phase의 기획 단계로 연결)

### 앱 스토어 관리

- 스토어 리뷰 모니터링 및 부정 리뷰 알림
- AI가 리뷰 답변 초안 작성 → 사람 확인 후 게시
- 평점 하락 시 원인 분석 및 긴급 대응 여부 판단

## KPI 추적

### 서비스 KPI

| 지표 | 측정 방법 | 목표 |
|------|----------|------|
| API 응답 시간 (p95) | 모니터링 대시보드 | 1초 이내 |
| 에러율 | 모니터링 대시보드 | 0.1% 이하 |
| 앱 크래시율 | Crashlytics | 0.5% 이하 |
| 웹 Core Web Vitals | Lighthouse | Good 등급 |
| 앱 스토어 평점 | 스토어 대시보드 | 4.0 이상 |

### AI 프로세스 KPI

12_quality_metrics와 연동하여 다음을 함께 추적한다:

- AI가 생성한 핫픽스의 성공률
- 피드백 → Jira 변환 정확도
- 모니터링 알림 오탐률

## 안정화 완료 기준

다음 조건을 모두 충족하면 안정화 구간을 종료한다:

- 에러율이 7일 연속 목표 이내
- 크래시율이 7일 연속 목표 이내
- 핫픽스 없이 7일 경과
- 주요 사용자 피드백 대응 완료

안정화 완료 후 일반 운영 모드로 전환한다.
