# Human Checkpoint

## 목적

AI가 만든 결과물을 사람이 검증하는 단계를 정의한다.
모든 작업에 동일한 수준의 검증을 요구하지 않고, risk_level에 따라 승인 방식을 차등 적용한다.

---

## 체크포인트 흐름

### Track A: 프로젝트 시작 (신규/리뉴얼)

```
프로젝트 정의 (사람)
    ↓
AI 전체 분석 + 기술 스택 + 시스템 설계
    ↓
CP0: 프로젝트 방향 승인 (사람)
    ↓
AI 기능 분해 → Epic / Story / Task 일괄 생성
    ↓
CP1: 전체 범위 확인 (사람)
    ↓
Sprint 단위로 Track B 반복
```

### Track B: 기능 개발 (공통)

```
Jira 티켓
    ↓
AI 요구사항 확장 → CP1: 요구사항 확인
    ↓
AI 작업 분해/계획 수립 → CP2: 계획 승인
    ↓
AI 테스트 설계 → CP2.5: 테스트 리뷰 (명세 정합성 검증)
    ↓
AI 코드 구현 (테스트 통과 목표)
    ↓
AI 리뷰 / PR 생성 → CP3: 구현 리뷰
    ↓
CI/CD → CP4: 배포 승인 (플랫폼별 차등)
    ↓
프로덕션 운영 / 모니터링 → CP5: 안정화 확인
```

---

## CP0: 프로젝트 방향 승인 (Track A 전용)

신규/리뉴얼 프로젝트에서 AI가 전체 분석과 기술 스택 결정을 완료한 후, 사람이 프로젝트 방향을 승인한다.

- 확인 대상: 기술 스택, 아키텍처 패턴, 프로젝트 구조, 리뉴얼 범위, 실행 일정
- 수준: **프로젝트에서 가장 중요한 의사결정** — 방향이 틀리면 전체가 틀어진다
- 거부 시: 사람이 방향을 수정하고 AI가 재분석

### 확인 항목

| 항목 | 내용 |
|------|------|
| 기술 스택 | AI가 선정한 언어, 프레임워크, DB가 적절한가 |
| 아키텍처 | 모놀리스/마이크로서비스, 모노레포/멀티레포 등 |
| 리뉴얼 범위 | 어디까지 새로 쓰고 어디를 유지하는가 |
| SDK/외부 연동 | 기존 SDK 유지/변경 방침이 맞는가 |
| 실행 계획 | 전체 일정과 단계별 목표가 현실적인가 |

---

## CP1: 요구사항 / 범위 확인

CP1은 Track에 따라 확인 대상이 다르다.

### Track A에서의 CP1: 전체 범위 확인

AI가 Epic/Story/Task를 일괄 생성한 후 사람이 전체 구조를 확인한다.

- 확인 대상: 기능 누락 여부, 우선순위 적절성, 범위 과잉/부족
- 수준: 전체 Jira 구조를 훑어보고 방향 확인
- 거부 시: 사람이 누락/수정 사항을 지시 → AI 재생성

### Track B에서의 CP1: 요구사항 확인

AI가 개별 Jira 티켓을 확장한 후 사람이 방향을 확인한다.

- 확인 대상: 요구사항, 인수 조건, 영향 범위
- 수준: 방향이 맞는지만 판단 ("훑어보고 OK")
- 거부 시: 사람이 코멘트 추가 → AI 재확장

---

## CP2: 계획 승인

AI가 작업 분해와 실행 계획을 수립한 후 사람이 승인한다.

- 확인 대상: 작업 분해, 의존성, 순서, 범위 적절성
- 과잉 설계나 누락을 방지하기 위한 단계

### risk_level별 승인 방식

| risk_level | 승인 방식 |
|:---:|---|
| low | 자동 승인 (스킵 가능) |
| medium | 알림 + 24시간 내 이의 없으면 자동 진행 |
| high | 사람이 확인 후 승인 |
| critical | 사람이 확인 + 다른 사람 교차 검토 |

---

## CP2.5: 테스트 리뷰

테스트 코드 작성 후, 기능 구현에 앞서 테스트가 명세를 정확히 반영하는지 검증한다.
테스트가 잘못되면 구현도 잘못된 방향으로 진행되므로, 구현 전에 테스트의 정확성을 먼저 확인한다.

- 확인 대상: 인수 조건 대비 테스트 커버리지, 엣지 케이스 누락, 테스트 설계 적절성
- AI가 1차 리뷰를 수행한다 (Review AI가 테스트 코드를 검증)
- 반려 시: Test AI가 테스트를 수정하고 재리뷰

### risk_level별 승인 방식

| risk_level | 승인 방식 |
|:---:|---|
| low | AI 리뷰만으로 통과 |
| medium | AI 리뷰만으로 통과 (결과를 사람에게 알림) |
| high | AI 리뷰 + 사람이 테스트 케이스 확인 |
| critical | AI 리뷰 + 사람이 테스트 케이스 + 테스트 코드 확인 |

### 검증 항목

| 항목 | 내용 |
|------|------|
| 인수 조건 커버리지 | 모든 인수 조건에 대응하는 테스트가 있는가 |
| 엣지 케이스 | 경계값, 빈 값, 에러 상황을 포함하는가 |
| 플랫폼 일관성 | iOS/Android 동일 기능의 테스트가 동일한 시나리오를 검증하는가 |
| 명세 정합성 | 테스트가 검증하는 동작이 명세와 일치하는가 (추측 기반 아닌가) |
| 독립성 | 테스트 간 순서 의존 없이 독립 실행 가능한가 |

## CP3: 구현 리뷰

**AI가 리뷰를 전부 수행한다** (Claude 1차 + Codex 2차 교차 리뷰).
사람에게는 AI가 "꼭 확인이 필요하다"고 판단한 항목만 전달한다. 해당 항목이 없으면 사람 확인 없이 통과한다.

### AI가 사람에게 전달하는 경우

- 가드레일 영역(DB, 결제, 인증, 보안, 인프라)에 변경이 감지된 경우
- AI가 판단하기 어려운 비즈니스 로직의 정확성
- iOS/Android 간 동작 차이가 감지되었으나 의도 여부를 모르는 경우
- 아키텍처에 큰 영향을 미치는 설계 변경

### AI가 직접 처리하는 것 (사람에게 전달 안 함)

- 코드 스타일, 포맷, 린트 (도구가 처리)
- 테스트 커버리지 (AI가 측정)
- 보안/성능 문제 (AI가 직접 반려)
- 플랫폼 필수 규칙 위반 (AI가 직접 반려)

상세 리뷰 체크리스트는 03_ai_dev_process를 참조한다.

---

## CP4: 배포 승인

CI/CD 통과 후 배포를 결정한다.

- 스테이징 배포: 자동 허용
- 프로덕션 배포: 사람 승인 필수
- 앱 스토어 제출: 사람 승인 필수

### 플랫폼별 배포 승인

| 플랫폼 | 스테이징 | 프로덕션 |
|:------:|:-------:|:-------:|
| 백엔드 | 자동 | 사람 승인 |
| 웹 | 자동 | 사람 승인 |
| 앱 (TestFlight / Internal) | 자동 | - |
| 앱 (스토어 제출) | - | 사람 승인 |
| 앱 (단계적 출시 확대) | - | 사람 승인 |

---

## CP5: 런칭 후 안정화 확인

프로덕션 배포 후 안정화 구간을 모니터링한다.

- 에러율, 크래시율, 응답 시간을 자동 추적
- 기준 초과 시 AI가 알림 + 원인 분석
- 핫픽스 필요 시 긴급 프로세스 진행 (18_post_launch 참조)
- 안정화 완료 기준 충족 시 사람이 안정화 종료를 확정한다
