# UI/UX 디자인 시안 생성

대상 화면: **$ARGUMENTS**

대상이 지정되지 않았으면 사용자에게 어떤 화면의 시안을 만들지 물어보세요.

## 프로세스 위치

이 스킬은 **Track A — A-7. 디자인 시안 (비주얼 방향 확정)** 단계에서 사용한다.

```
CP1 (전체 범위 확인) 완료
    ↓
/design-proposal 홈화면      ← 이 스킬
/design-proposal 카드상세
/design-proposal ...
    ↓
사람이 비주얼 방향 확인 → 승인
    ↓
승인된 시안이 디자인 시스템 기준이 됨
    ↓
Sprint 시작 — 개별 티켓은 이 기준을 따라 구현
```

**Track B에서도 사용 가능**: 새로운 UX 패턴 도입 시 Designer AI 1차 단계에서 호출.

---

## 핵심 원칙

1. **구조가 달라야 한다** — 색상·톤만 다른 변형은 시안이 아니다. 각 시안은 레이아웃, 정보 계층, 인터랙션 패턴이 달라야 한다.
2. **디자이너 수준 퀄리티** — 실제 적용 가능한 수준의 완성도. 정렬, 여백, 타이포그래피, 색상 조화에 신경 쓴다.
3. **모카키 맥락 반영** — 출입 앱이라는 서비스 특성, 고령/비기술 사용자 타겟, iOS/Android 양 플랫폼 고려.
4. **접근성 준수** — 색상 대비(WCAG AA 4.5:1 이상), 최소 터치 영역(44×44pt), 본문 최소 14pt.

## 시안 생성 프로세스

### 1단계: 맥락 파악
- `flow/renewal.md`를 읽어 해당 화면의 개선 방향 확인
- `모카키_기능분석_리뉴얼사전조사.md`에서 현행 화면 구조 확인
- 기존 디자인 토큰이 있으면 (`docs/design-tokens.md` 또는 `artifacts/00_init/design_concept.md`) 반드시 참조
- 필요 시 iOS 소스코드에서 현행 UI 요소 확인 (ViewModel, XIB)

### 2단계: 구조 방향 결정
최소 3개 이상의 **구조적으로 다른** 방향을 설계한다. 아래는 방향 구분 기준:

**레이아웃 차이 (필수 — 최소 하나는 달라야 함)**
- 카드 중심 vs 리스트 중심 vs 그리드
- 단일 포커스 vs 분할 화면 vs 탭 분리
- 풀스크린 카드 vs 컴팩트 카드 vs 타일
- 고정 헤더 + 스크롤 vs 전체 스크롤 vs 섹션별 접기

**정보 계층 차이**
- 핵심만 (이름 + 상태) vs 요약 (+ 구성원 수, 장치) vs 전체 (모든 정보)
- 숨김/펼침 구조 vs 항상 표시 vs 탭 전환

**인터랙션 차이**
- 탭 → 상세 vs 스와이프 vs 롱프레스
- 하단 시트 vs 전체 화면 전환 vs 인라인 확장

### 3단계: 상태별 화면 생성
각 시안마다 해당 화면의 주요 상태를 모두 표현:
- 예: 홈 화면이면 카드 없음 / 1장 / 2장+ / 오류 상태
- 예: 카드 상세면 활성 / 정지 / 만료 상태

### 4단계: 화면 플로우 (선택)
화면이 다른 화면과 연결되는 경우, 전환 흐름을 시각적으로 표현한다:
- 화면 간 이동 방향 (예: 홈 → 카드 상세 → 문열기 결과)
- 전환 방식 (push / modal / bottom sheet)
- 뒤로가기 경로

단일 화면 시안만 요청한 경우 생략 가능. `--flow` 옵션으로 명시적 요청 가능.

### 5단계: 시안 생성

**도구 우선순위:**

#### 방법 1: Google Stitch MCP (권장)

Stitch MCP 서버가 연결되어 있으면 이를 우선 사용한다. 토큰 소모가 적고 디자인 퀄리티가 높다.

```
Stitch MCP 호출:
- 각 시안 방향별로 별도 프롬프트 생성
- 프롬프트에 포함할 내용: 화면 목적, 레이아웃 방향, 상태별 화면, 플랫폼(iOS/Android), 디자인 토큰
- 출력 포맷: HTML 또는 SwiftUI/Flutter (구현 플랫폼에 따라)
- 결과를 산출물 경로에 저장
```

프롬프트 예시:
```
"모바일 앱 [화면명] 화면.
방향: [레이아웃 설명].
상태: [상태 목록].
스타일: 미니멀, Noto Sans KR, 16px 본문, 44px 최소 터치 영역.
[추가 맥락]"
```

#### 방법 2: Agent HTML 생성 (Stitch 불가 시 폴백)

Stitch MCP가 없거나 실패하면 Agent로 HTML을 직접 생성한다.

산출물 경로에 HTML 파일을 저장한다.

## HTML 작성 규칙

### 폰 프레임 — 멀티 해상도

시안은 **3가지 디바이스 크기**로 생성하여 레이아웃이 모든 비율에서 동작하는지 검증한다.

| 디바이스 | 프레임 크기 | 비율 | 용도 |
|---------|-----------|------|------|
| **소형** (iPhone SE / Galaxy A) | 250×444px | 16:9 | 작은 화면에서 하단 요소 잘림, 터치 영역 확보 검증 |
| **표준** (iPhone 15 / Galaxy S24) | 280×606px | 19.5:9 | 기준 화면. 모든 시안의 기본 |
| **대형** (iPhone 15 Pro Max / Galaxy S24 Ultra) | 300×650px | 19.5:9 | 큰 화면에서 여백/확장 활용 검증 |

기본은 **표준** 프레임으로 생성하고, `--responsive` 옵션 시 3가지 모두 나란히 표시한다.

```
표준 프레임:
width: 280px, height: 606px
border-radius: 40px, border: 5px solid #1c1c1e
Dynamic Island: 90×24px, radius 14px
Status bar: 38px height
```

주의사항:
- 소형 화면에서 하단 버튼/탭바가 콘텐츠와 겹치지 않는지 확인
- 대형 화면에서 카드/리스트가 과도하게 늘어나지 않는지 확인
- 가로 스크롤이 발생하지 않는지 확인

### 글꼴 크기 확대 대응 (Dynamic Type / 접근성)

사용자가 시스템 설정에서 글꼴을 크게 설정했을 때도 레이아웃이 정상 동작해야 한다.
`--large-text` 옵션 시 **기본 / 확대(×1.5) / 최대(×2.0)** 3단계를 나란히 표시한다.

| 단계 | 배율 | iOS 설정 | Android 설정 | 검증 포인트 |
|------|------|---------|-------------|------------|
| **기본** | ×1.0 | Default | 기본값 | 기준 화면 |
| **확대** | ×1.5 | Accessibility Text Sizes (XXXL) | 글꼴 크기 '최대' | 대부분의 접근성 사용자 |
| **최대** | ×2.0 | Largest Accessibility Size | 표시 크기 + 글꼴 '최대' | 극단적 케이스 |

**시안 설계 원칙:**
- 텍스트 영역은 **고정 높이 금지** — 글꼴이 커져도 텍스트가 잘리지 않도록 가변 높이 사용
- 긴 텍스트는 말줄임(...) 대신 **줄바꿈** 허용 (단, 최대 2~3줄 제한)
- 아이콘 + 텍스트 조합은 글꼴 확대 시 **세로 스택으로 전환** 가능하도록 설계
- 버튼 텍스트가 길어져도 버튼 크기가 함께 늘어나도록 (고정 width 금지)
- 카드/셀 높이는 콘텐츠에 따라 자동 조절

**구현 시 플랫폼 규칙:**

| 플랫폼 | 방법 | 주의 |
|--------|------|------|
| iOS | `UIFontMetrics` / SwiftUI `.dynamicTypeSize` | `minimumScaleFactor` 남용 금지 — 글씨를 줄이면 접근성 무효화 |
| Android | `sp` 단위 사용 (dp 아닌 sp) | `android:maxLines` + `ellipsize` 대신 wrap 허용 |

### 컬러 모드
- `--dark` 옵션이 있으면 다크 모드로 생성
- 옵션이 없으면 각 시안의 성격에 맞게 라이트/다크 자유 선택
- 동일 구조의 라이트/다크 변형은 1개 시안으로 취급 (구조가 같으므로)

### 탭바
리뉴얼 기준 3탭: 홈 | 가이드 | 내정보

### 폰트

시안 단계에서는 웹 폰트를 사용하되, 실제 구현 시 플랫폼 전환 규칙을 따른다.

**시안용 (HTML):**
```css
font-family: 'Noto Sans KR', sans-serif;
@import url('https://fonts.googleapis.com/css2?family=Noto+Sans+KR:wght@300;400;500;600;700;800&display=swap');
```

**실제 구현 시 플랫폼별 전환:**

| 용도 | 시안 (HTML) | iOS 구현 | Android 구현 |
|------|-----------|----------|-------------|
| 시스템 폰트 | Noto Sans KR | SF Pro (.system) | Roboto (기본) |
| 한글 보조 | Noto Sans KR | Apple SD Gothic Neo (시스템 내장) | Noto Sans KR (시스템 내장) |
| 숫자/영문 강조 | — | SF Pro Rounded (선택) | Google Sans (선택) |

**타입 스케일 (디자인 토큰으로 관리):**

| 용도 | 크기 | 무게 | 행간 |
|------|------|------|------|
| 대제목 (Hero) | 28px | 700 (Bold) | 1.3 |
| 제목 (Title) | 20px | 600 (SemiBold) | 1.4 |
| 부제목 (Subtitle) | 17px | 500 (Medium) | 1.4 |
| 본문 (Body) | 16px | 400 (Regular) | 1.5 |
| 캡션 (Caption) | 13px | 400 (Regular) | 1.4 |
| 최소 (Min) | 11px | 400 (Regular) | 1.3 |

- 본문 최소 16px (고령 사용자 가독성)
- iOS Dynamic Type / Android 접근성 글꼴 크기 대응 고려
- 승인 후 디자인 토큰에 확정값 기록

### 접근성
- 텍스트/배경 색상 대비: WCAG AA 기준 4.5:1 이상
- 버튼/터치 영역: 최소 44×44pt
- 본문 텍스트: 최소 14pt (고령 사용자 고려)
- 상태 표시: 색상만이 아닌 아이콘/텍스트 병행 (색각 이상 대응)

### 텍스트
- renewal.md에 정의된 문구 사용
- 없으면 서비스 맥락에 맞게 작성하되, 기술 용어 지양
- 예시 데이터: 홍길동, 모카 아파트, 우리집 등 일관되게 사용

### 시안 라벨
각 시안 위에 구조적 특징을 한 줄로 표기:
```
예: "카드 풀스크린 + 하단 요약" / "타일 리스트 + 인라인 확장" / "대시보드 + 퀵 액션"
```

### 비교표
파일 하단에 시안별 비교 요약 테이블 필수 포함:
- 레이아웃 방식
- 카드 표현
- 정보 밀도
- 타겟 사용자
- 강점 / 약점

## 옵션

- `--count=N`: 시안 개수 (기본 3)
- `--dark`: 다크 모드 전용
- `--light`: 라이트 모드 전용
- `--state=X`: 특정 상태만 (예: --state=empty)
- `--flow`: 화면 간 전환 플로우도 포함
- `--tokens`: 시안과 함께 디자인 토큰을 바로 추출
- `--responsive`: 소형/표준/대형 3가지 프레임을 나란히 표시하여 해상도별 레이아웃 검증
- `--large-text`: 기본/확대(×1.5)/최대(×2.0) 글꼴 크기 3단계를 나란히 표시하여 Dynamic Type 대응 검증
