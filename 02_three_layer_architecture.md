# 02. 3-Layer 아키텍처

> **"프로세스는 하나, 실행 방법은 여러 개, 진입점은 역할별"**

---

## 1. 개요 — 왜 3-Layer인가

### 문제

현재 28개 프로세스 문서는 규칙·실행·인터페이스가 혼재되어 있다.
"Planner 에이전트는 이런 역할"(규칙)과 "Claude Code CLI에서 `/plan`을 실행"(실행/인터페이스)이 같은 문서에 섞여, 도구를 바꾸면 프로세스 전체를 다시 정의해야 하는 구조다.

### 해결

3개 레이어로 분리하여 **도구에 종속되지 않는 프로세스 플랫폼**을 구축한다.

| 레이어 | 역할 | 변경 빈도 |
|--------|------|----------|
| **Layer 1: 프로세스 정의** | 무엇을, 어떤 순서로, 어떤 기준으로 | 거의 불변 |
| **Layer 2: 실행 엔진** | 어떤 도구로, 어떤 환경에서 실행 | 환경 추가 시 변경 |
| **Layer 3: 인터페이스** | 누가, 어떻게 진입 | 역할/채널 추가 시 변경 |

### 효과

- **Layer 1만 읽으면** 프로세스 전체를 이해할 수 있다
- **Layer 2만 교체하면** 실행 도구를 바꿀 수 있다 (다른 AI, 다른 CI)
- **Layer 3만 추가하면** 새 진입점을 만들 수 있다 (웹 UI, 챗봇)

## 3. Layer 1: 프로세스 정의 (Process Definition)

> 프로세스의 핵심 규칙을 정의한다.
> 현재 기본 도구 스택: **Claude Code** (AI 에이전트) + **Jira** (이슈 관리) + **GitHub** (코드 관리).
> 선택 옵션: **Teams** (추가 알림/긴급 알림).

### 3.1 파이프라인 흐름

#### Track A — 프로젝트 시작 (그린필드/리뉴얼)

```
아이디어 입력
  → CP0: 프로젝트 방향 승인
  → 전체 분석 (기존 시스템, 기술 스택, 리스크)
  → 기능 분해 (Epic → Story → Task)
  → 플랫폼별 태스크 분리
  → Jira에 카드 자동 생성
  → CP1: 전체 범위 승인
  → Track B로 전환 (카드 단위 개발)
```

#### Track B — 티켓 단위 개발 (운영 서비스)

```
티켓 할당
  → 요구사항 분석 (Planner)
  → CP1: 요구사항 확인
  → 설계 문서 작성 (Spec Writer)
  → CP2: 설계 승인 (risk_level별 차등)
  → 테스트 설계 (Tester)
  → Test Validator 자동 검증 (V-1~V-12)
  → CP2.5: 테스트 리뷰
  → 코드 구현 (Developer)
  → 코드 리뷰 (Claude 1차 + Codex 2차 교차 리뷰)
  → CP3: 구현 리뷰
  → 릴리스 준비 (Release Manager)
  → CP4: 배포 승인
  → 배포 후 안정화
  → CP5: 안정화 확인
```

#### 복잡도 vs risk_level — 프로세스 경량화

프로세스 비용(토큰, 시간)을 최적화하기 위해, 작업 규모에 따라 단계를 줄인다.

**복잡도** (`/idea`에서 사용 — 기획 Phase):
아이디어 텍스트를 보고 오케스트레이터가 판별한다.

| 복잡도 | 조건 | 기획 프로세스 | 예상 토큰 |
|--------|------|-------------|----------|
| 단순 | 단일 플랫폼, 기존 화면 수정, API 변경 없음 | Ideator+Planner 통합 → Spec → Jira | ~100K |
| 보통 | 단일 플랫폼 신규 기능, API 변경 포함 | Ideator → Planner → Spec → 교차 리뷰(통합) | ~250K |
| 복잡 | 멀티 플랫폼, 새 기술/SDK/하드웨어 | 풀 프로세스 (PoC/디자인 시안 포함) | ~400K |

**risk_level** (`/in-progress`에서 사용 — 개발 Phase):
Jira 티켓의 메타 필드에서 읽는다. 없으면 medium 기본값.

| risk_level | 조건 | 개발 프로세스 | 예상 토큰 |
|-----------|------|-------------|----------|
| low | 기존 파일 수정만, 신규 없음 | 테스트→구현→리뷰 1회 | ~150K |
| medium | 신규 파일 있으나 기존 패턴 | 테스트+Validator→구현→리뷰 1회 | ~400K |
| high/critical | 멀티 플랫폼, 새 기술, 인증/결제 | 풀 프로세스 (Validator+교차 리뷰+1차+2차) | ~800K |

**Agent 모델 최적화**: 단순/low 작업에는 Haiku/Sonnet을, 복잡/high 작업에는 Opus를 사용한다.
상세: `/idea` → idea.md, `/in-progress` → in-progress.md 참조.

#### Track A → Track B 전환점

- CP1 승인 완료 시점에 Track A 산출물(기능 분해 결과)이 개별 티켓으로 전환된다
- 각 티켓은 독립적으로 Track B 파이프라인을 탄다
- 플랫폼 간 의존성이 있는 경우, API 명세 기반으로 순서를 조정한다

### 3.2 에이전트 역할 정의

8개 전문 에이전트가 순차적으로 산출물을 생성한다. 각 에이전트는 선행 산출물을 입력받고, 정해진 형식의 산출물을 출력한다.

#### 파이프라인 에이전트 (순차 실행)

| # | 에이전트 | 입력 | 출력 | 금지 행위 |
|---|---------|------|------|----------|
| 1 | **Planner** | 티켓 정보 | 요구사항 분석서, 작업 분해 | 코드 작성, 설계 결정 |
| 2 | **Spec Writer** | 요구사항 분석서 | 설계 문서 (API/DB/구조) | 코드 작성, 테스트 작성 |
| 3a | **Designer (1차)** | 설계 문서 | UI/UX 와이어프레임, 프로토타입 | 코드 작성 (조건부: UI 변경 있을 때만) |
| 3b | **Designer (2차)** | 구현 코드, 1차 와이어프레임 | 비주얼 스펙 (구현 반영) | 코드 작성 (조건부: UI 변경 있을 때만) |
| 4 | **Tester** | 설계 문서, 인수 조건 | 테스트 케이스, 테스트 코드 | 구현 코드 작성 |
| 5 | **Developer** | 설계 문서, 테스트 코드 | 구현 코드, 변경 요약 | 테스트 수정, 설계 변경 |
| 6 | **Reviewer (Claude)** | 구현 코드, 설계 문서 | 1차 리뷰 결과 (통과/반려/에스컬레이션) | 코드 직접 수정 |
| 7 | **Release Manager** | 리뷰 통과 코드 | 릴리스 노트, 배포 체크리스트 | 코드 수정 |

> **Cross Reviewer (Codex)**: Claude 리뷰와 독립적으로 동일 코드를 리뷰한다. 두 AI의 결과를 종합하여 최종 리뷰 판정.

#### 상시 에이전트 (다른 에이전트가 호출)

| # | 에이전트 | 역할 | 입력 | 출력 | 금지 행위 |
|---|---------|------|------|------|----------|
| 8 | **Domain Expert** | 도메인 지식 관리·검증 | 도메인 질문, 산출물 내 용어 | 도메인 판단 결과, 용어 검증, 지식 문서 갱신 | 코드 작성, 설계 결정 |

**Domain Expert 상세**:

- **도메인 지식 문서 관리**: 프로젝트 초기에 `domain_knowledge.md` 초안 작성 (용어집, 비즈니스 규칙, 사용자 플로우, 업계 규제)
- **도메인 질문 응답**: 다른 에이전트가 도메인 판단이 필요할 때 호출하여 답변 (예: "이 기능에서 '키'란 물리키인가 모바일키인가?")
- **산출물 용어 검증**: Planner, Spec Writer의 산출물에서 도메인 용어가 올바르게 사용되었는지 검증
- **지식 축적**: 개발 과정에서 발견된 새로운 도메인 지식을 문서에 반영 (자동 갱신)
- **입력 소스**: 기존 코드베이스, 기획 문서, 레거시 시스템 분석, 사람의 보정

> Domain Expert는 파이프라인에 고정 순서로 참여하지 않는다.
> Planner가 요구사항 분석 중 도메인 모호성을 발견하면 호출하고,
> Spec Writer가 API 네이밍 시 업계 용어를 확인할 때 호출하는 식으로 **필요 시 호출**된다.

**에이전트 간 데이터 흐름**:
```
Planner → Spec Writer → Designer(1차) → Tester → Developer → Designer(2차) → Claude Review + Codex Review → Release Manager
            ↓                              ↓          ↑
        설계 문서 ──────────────────→ 테스트 코드 ───┘
                                    (테스트 선행 원칙)
    ↑               ↑
    └── Domain Expert (필요 시 호출) ──┘
```

### 3.3 체크포인트 (CP0 ~ CP5)

각 체크포인트는 사람의 확인이 필요한 게이트다.

| CP | 시점 | 검증 내용 | 승인 기준 |
|----|------|----------|----------|
| **CP0** | Track A 시작 | 프로젝트 방향, 목표, 범위 | 기획자/PM 승인 |
| **CP1** | 요구사항 완료 | 기능 범위, 우선순위, 제약 조건 | Track A: 전체 범위 / Track B: 티켓 요구사항 |
| **CP2** | 설계 완료 | 설계 문서 품질, 기술 결정 | risk_level별 차등 (아래 참조) |
| **CP2.5** | 테스트 설계 완료 | 인수 조건 전수 매핑, 5개 카테고리 커버리지 | risk_level별 차등 (아래 참조) |
| **CP3** | 구현 완료 | 코드 품질, 요구사항 충족, 가드레일 준수 | AI 리뷰 통과 (가드레일 위반 시 에스컬레이션) |
| **CP4** | 배포 직전 | 스테이징 검증 완료, 배포 준비 | 스테이징: 자동 / 프로덕션: 사람 승인 |
| **CP5** | 배포 후 | 안정화 지표 (에러율, 성능) | 기준치 이내 확인 |

**risk_level별 CP2.5 승인 매트릭스**:

| risk_level | 승인 방식 |
|-----------|----------|
| low | AI 리뷰만으로 자동 승인 |
| medium | AI 리뷰 후 알림, 24시간 묵시적 승인 |
| high | AI 리뷰 + 사람이 테스트 케이스 직접 확인 |
| critical | AI 리뷰 + 사람이 테스트 케이스·시나리오 전수 확인 |

**risk_level별 CP2 승인 매트릭스**:

| risk_level | 승인 방식 |
|-----------|----------|
| low | 자동 승인 (알림만) |
| medium | 알림 후 24시간 묵시적 승인 |
| high | 담당자 리뷰 필수 |
| critical | 담당자 + 교차 리뷰 |

**risk_level 판정 기준**:

| risk_level | 조건 |
|-----------|------|
| **critical** | 결제·인증·보안·개인정보 영역 변경, DB 스키마 마이그레이션 |
| **high** | 공용 API 변경, 핵심 비즈니스 로직 수정, 성능 임계 영역 |
| **medium** | 기존 기능 수정, UI 변경, 의존성 업데이트 |
| **low** | 신규 기능 추가 (기존 코드 영향 없음), 문서·설정 변경, 테스트 추가 |

> Planner가 티켓 분석 시 risk_level을 산정하고, CP1에서 확인받는다.

### 3.4 가드레일

AI가 단독으로 변경할 수 없는 영역. 해당 영역 변경이 감지되면 자동으로 에스컬레이션된다.

**AI 단독 변경 불가 영역**:
- 데이터베이스 스키마 마이그레이션
- 결제 시스템
- 인증/인가 시스템
- 보안 설정
- 인프라/네트워크 설정

**감지 방식**: 파일 경로 패턴 매칭 (`migrations/**`, `payment/**`, `auth/**`, `security/**`, `infra/**`)

**에스컬레이션 조건**:
- 가드레일 영역 변경 시 → 담당 시니어 개발자에게 알림
- 리뷰어 간 판정 불일치 시 → 기술 리드에게 에스컬레이션
- 3회 이상 CP 반려 시 → 프로세스 재검토

> 참조: [07_ai_dev_guardrails.md](07_ai_dev_guardrails.md), [09_escalation_policy.md](09_escalation_policy.md)

### 3.5 AI 역질문 프로세스

에이전트는 입력이 불충분하면 **추측하지 않고 질문**한다.

**역질문 조건**:
- 인수 조건이 모호하거나 누락된 경우
- 설계 결정에 필요한 정보가 부족한 경우
- 상충하는 요구사항이 감지된 경우

**처리 흐름**:
```
에이전트가 입력 부족 감지
  → 작업 중단
  → 질문 목록을 Jira 코멘트에 작성
  → 티켓 상태를 "Blocked" 로 전환
  → 답변 수신 후 작업 재개
```

**원칙**: 불완전한 입력으로 산출물을 생성하는 것보다, 질문으로 명확히 하는 것이 낫다.

### 3.8 실패·재시도·타임아웃 정책

#### 에이전트 실행 실패 시

| 실패 유형 | 처리 |
|----------|------|
| AI 응답 오류 (API 에러, 타임아웃) | 최대 2회 자동 재시도 → 실패 시 해당 단계 중단, Jira에 실패 기록 |
| 산출물 형식 검증 실패 | 에이전트에게 검증 오류 내용 전달 후 1회 재생성 → 실패 시 에스컬레이션 |
| 가드레일 위반 감지 | 재시도 없이 즉시 에스컬레이션 |

#### CP 승인 타임아웃

| risk_level | 타임아웃 | 타임아웃 시 처리 |
|-----------|---------|----------------|
| low | 없음 (자동 승인) | - |
| medium | 24시간 | 묵시적 승인 처리 |
| high | 72시간 | 담당자 + 리드에게 리마인더 알림 |
| critical | 없음 | 승인 전까지 대기 (리마인더는 24시간마다) |

#### 동시 작업 충돌

- 같은 레포에서 여러 티켓이 동시 진행될 수 있다
- **원칙**: 각 티켓은 독립 브랜치에서 작업 → PR 단계에서 충돌 해결
- 동일 파일을 수정하는 티켓이 감지되면 Jira에 경고 코멘트 추가
- 머지 충돌 발생 시: AI가 자동 해결 시도 → 실패 시 개발자에게 에스컬레이션

### 3.9 산출물 버전 관리

- **로컬 CLI 모드**: 산출물은 해당 레포의 `artifacts/{ticket_id}/` 에 생성, 작업 브랜치에 커밋
- **서버/봇 모드**: 산출물을 작업 브랜치에 커밋 후 push (봇 계정 사용)
- **멀티 레포 시**: 각 플랫폼 레포의 `artifacts/` 에 해당 플랫폼 산출물 저장
- **플랫폼 공통 산출물** (요구사항 분석서 등): 중앙 레포 또는 원본 티켓이 속한 레포에 저장
- 산출물 수정 이력은 git 히스토리로 추적

### 3.10 롤백 정책

| 상황 | 처리 |
|------|------|
| CP 반려 | 해당 단계 에이전트 재실행 (피드백 반영), 최대 3회 → 에스컬레이션 |
| 배포 후 장애 (CP5 실패) | hotfix 플로우 진입 (축약된 Track B: plan → dev → review) |
| 긴급 롤백 필요 | 이전 버전으로 즉시 롤백 (사람 판단), 원인 분석은 별도 티켓으로 |

### 3.11 프로젝트 초기 설정

새 프로젝트가 이 프로세스를 사용하려면 다음을 정의해야 한다.

**필수 설정**:

| 항목 | 설명 | 예시 |

> **도메인 지식 문서 (`domain_knowledge.md`)**:
> - 프로젝트 시작 시 Domain Expert 에이전트가 기존 코드베이스·기획 문서를 분석하여 초안 작성
> - 이후 개발 과정에서 발견되는 새 도메인 지식을 Domain Expert가 자동 갱신
> - 사람이 주기적으로 검토·보정 (특히 업계 규제, 비즈니스 정책 등 AI가 추측하면 안 되는 영역)
>
> **포함 내용**:
> - 용어집 (프로젝트 고유 용어와 정의)
> - 비즈니스 규칙 (예: "만료된 키는 삭제가 아니라 비활성 처리")

> - 업계 규제·제약 (보안 인증, 프로토콜 제약 등)
> - 레거시 히스토리 (왜 현재 구조가 이렇게 되었는지)

**설정 파일 위치**:
- 로컬 CLI: 각 레포의 `CLAUDE.md` 에 프로젝트 컨텍스트로 기록
- 서버/봇: 오케스트레이터 설정 DB 또는 설정 파일

## 4. Layer 2: 실행 엔진 (Execution Engine)

> Layer 1의 규칙을 실제로 실행하는 방법.
> 현재 기본 실행 환경: **Claude Code CLI** (로컬) + **GitHub Actions** (CI/CD) + **Jira 웹훅 서버** (자동화).

### 4.1 실행 환경 옵션

| 환경 | 적합한 상황 | 구성 요소 |
|------|------------|----------|
| **로컬 CLI** | 개발자가 직접 실행 | Claude Code + CLAUDE.md + commands/ |
| **CI/CD** | 자동화 파이프라인 | GitHub Actions + Claude API |
| **서버/봇** | Jira 연동 자동 실행 | API 서버 + Jira 웹훅 + Claude API |

**환경 선택 기준**:
- 개발자 1인이 티켓 하나를 진행 → 로컬 CLI
- PR 생성 시 자동 리뷰 → CI/CD
- 기획자가 아이디어를 내면 자동 분석 → 서버/봇
- 3가지를 조합하여 사용 가능 (예: 기획은 서버/봇, 개발은 로컬 CLI)

### 4.2 오케스트레이터

에이전트의 순차 실행, 상태 관리, 산출물 검증을 담당한다.

**역할**:
- 티켓 상태 관리 (현재 단계, 다음 단계)
- 에이전트 순차 실행 및 산출물 전달
- 체크포인트 트리거 (CP 도달 시 알림 발송)
- 산출물 형식 검증 (Layer 1 규칙 준수 여부)
- 실패 시 재시도 / 에스컬레이션

**상태 머신**:
```
idle → planning → spec → design(1) → testing → developing → design(2) → reviewing → releasing → done
                                                                              ↓
                                                                         rejected → (이전 단계로)
```

**구현 옵션**:

| 모드 | 오케스트레이터 실체 | 상태 저장 | 적합한 팀 |
|------|-------------------|----------|----------|
| 수동 | **사람** (커맨드를 순차 실행) | Jira 티켓 상태 + 로컬 산출물 | 소규모, 학습 단계 |
| 반자동 | **GitHub Actions** (PR/커밋 이벤트 → 다음 단계 트리거) | Jira + GitHub | 중규모, 개발팀 |
| 자동 | **웹훅 수신 서버** (Jira 이벤트 → 전체 흐름 관리) | 서버 DB + Jira | 대규모, 성숙한 프로세스 |

> **도입 순서 권장**: 수동 → 반자동 → 자동 순으로 점진적으로 확장한다.
> 처음부터 자동 모드를 구축하지 않는다. 수동으로 프로세스를 검증한 후 자동화한다.

> 참조: [04_ai_agent_architecture.md](04_ai_agent_architecture.md) — 상태 테이블

### 4.3 에이전트 실행

**프롬프트 구조**:
```
시스템 프롬프트 = Layer 1 규칙 + 해당 에이전트 역할 정의 + 프로젝트 컨텍스트
```

- Layer 1 규칙: 파이프라인 흐름, 산출물 형식, 가드레일
- 역할 정의: 에이전트별 입출력, 금지 행위 (§3.2 참조)
- 프로젝트 컨텍스트: 기술 스택, 코드베이스 구조, 기존 산출물

**에이전트 간 산출물 전달**:
- 선행 에이전트의 산출물 파일 경로를 후행 에이전트에게 전달
- `artifacts/{ticket_id}/{stage}/` 경로 규칙 준수
- 산출물 형식 검증 후 다음 에이전트 실행

**멀티 에이전트 협업**:
- 플랫폼별 병렬 실행 가능 (예: iOS Tester + Android Tester 동시 실행)
- 리뷰 단계: Claude 리뷰 + Codex 리뷰 병렬 실행 후 종합

> 참조: [15_agent_prompts_guide.md](15_agent_prompts_guide.md)

### 4.4 CP 승인 메커니즘

체크포인트 승인은 Jira 워크플로우로 구현한다.

**승인 흐름**:
```
에이전트가 CP 도달
  → Jira 티켓 상태를 "Review" 로 전환 + 산출물 요약 코멘트 작성
  → 승인 담당자가 Jira에서 확인
  → "Approve" 트랜지션 클릭 → 다음 단계 자동 진행
  → "Reject" 트랜지션 클릭 + 반려 사유 코멘트 → 해당 단계 재실행
```

**구현 방식별 차이**:

| 모드 | CP 트리거 | 승인 감지 |
|------|----------|----------|
| 수동 | 사람이 `status`로 확인 후 다음 커맨드 실행 | 사람이 직접 판단 |
| 반자동 | GitHub Actions가 Jira 상태 변경 | Jira 웹훅으로 감지 |
| 자동 | 서버가 Jira 상태 자동 전환 | Jira 웹훅으로 감지 → 다음 에이전트 자동 실행 |

**묵시적 승인 (medium risk)**:
- 오케스트레이터가 CP 도달 시각을 기록
- 24시간 후 응답이 없으면 자동으로 "Approve" 트랜지션 실행
- 자동 모드에서만 동작. 수동 모드에서는 사람이 판단

### 4.5 외부 연동

| 연동 대상 | 용도 | 연동 방식 |
|----------|------|----------|
| **Jira** | 티켓 조회/생성/상태변경 | MCP Server (jira-mcp), REST API, 웹훅 |
| **GitHub** | PR 생성, 코드 조회, 브랜치 관리 | `gh` CLI, REST API, GitHub Actions |
| **Teams** (선택) | 추가 알림, 긴급 알림 | Teams 웹훅, Power Automate |
| **Figma / Google Stitch** | 디자인 스펙 조회, AI 디자인 생성 | Figma API, Stitch CLI/MCP 연동 |
| **모니터링** (Datadog 등) | 배포 후 안정화 지표 | API, 대시보드 |

### 4.6 Jira 프로젝트 구조

기획자의 시작점은 Jira다. 두 가지 진입 방식을 모두 지원한다.

#### 방식 A: 중앙 프로젝트에서 시작

기획 전용 Jira 프로젝트(예: `AI-DEV`)를 두고, 여기서 아이디어를 생성한다.

```
기획자: [AI-DEV] 프로젝트에 Idea 생성
        → "신규 장치 A 추가"
    ↓
AI가 분석 → 기능 분해 → 플랫폼별 분리
    ↓
각 플랫폼 프로젝트에 티켓 자동 생성:
    → [SERVER] MOCA-101  "장치 A API 추가"
    → [IOS]    MOCA-102  "장치 A 위젯 추가"
    → [ANDROID] MOCA-103 "장치 A 위젯 추가"
    ↓
[AI-DEV] 원본 이슈에 하위 티켓 링크 연결
```

**장점**: 기획자가 플랫폼을 몰라도 된다. 전체 진행 상태를 한 프로젝트에서 조회 가능.

#### 방식 B: 플랫폼 프로젝트에 직접 생성

기획자가 특정 플랫폼 프로젝트에 직접 이슈를 생성한다.

```
기획자: [SERVER] 프로젝트에 Idea 생성
        → "신규 장치 A API 추가"
    ↓
AI가 분석 → 연관 플랫폼 감지
    ↓
관련 프로젝트에 연쇄 티켓 자동 생성:
    → [IOS]    MOCA-102  "장치 A 위젯 추가"
    → [ANDROID] MOCA-103 "장치 A 위젯 추가"
    ↓
각 티켓 간 링크 연결 + 의존성 설정
```

**장점**: 기획자가 이미 대상 플랫폼을 알 때 빠르게 진입 가능.

#### AI 판단 기준

AI는 아이디어 내용을 분석하여 자동으로 판단한다:
- 관련 플랫폼 목록 (서버/iOS/Android/웹/펌웨어)
- 플랫폼 간 의존성 순서
- 플랫폼별 태스크 범위

판단 결과는 CP0 승인 시 기획자에게 확인받는다.

### 4.7 멀티 레포 작업

하나의 Jira 아이디어가 여러 GitHub 레포(서버, iOS, Android, 웹, 펌웨어)에 걸칠 수 있다.

**처리 방식**:
1. Track A에서 기능 분해 시 **플랫폼별 태스크 분리** (§4.6의 Jira 프로젝트 구조에 따라)
2. 각 태스크에 대상 레포 태그 부여 (`[backend]`, `[ios]`, `[android]`, `[web]`, `[firmware]`)
3. 레포 간 의존성은 **API 명세 기반**으로 관리
   - 서버 API 변경 → 클라이언트 태스크에 선행 조건으로 연결
4. 각 레포는 독립적으로 Track B 파이프라인 실행
5. 필요 시 레포별 `CLAUDE.md` 유지

**의존성 순서 (일반적)**:
```
서버 API 명세 확정 → 서버 구현 → 클라이언트 구현 (병렬: iOS / Android / Web)
                                    ↑
                              펌웨어 프로토콜 확정 (해당 시)
```

### 4.8 환경별 구성 파일

| 환경 | 필요 파일 | 설명 |
|------|----------|------|
| **로컬 CLI** | `CLAUDE.md` | 프로젝트 컨텍스트 (Layer 1 규칙 참조) |
| | `commands/*.md` | 커맨드 정의 (Layer 3 인터페이스) |
| | `agents/*.md` | 에이전트 프롬프트 |
| | `mcp.json` | 외부 연동 설정 |
| **CI/CD** | `.github/workflows/` | 워크플로우 정의 |
| | 시스템 프롬프트 파일 | Layer 1 규칙 + 에이전트 역할 |
| **서버** | 오케스트레이터 코드 | 상태 관리, 에이전트 호출 |

| | 환경 설정 | 연동 키, 엔드포인트 URL |

## 5. Layer 3: 인터페이스 (Interface)

> 사람이 프로세스에 진입하는 방법. 역할에 따라 진입점이 다르다.
> 기본 인터페이스: **Claude Code CLI** (개발자) + **Jira** (기획자/리뷰어).
> 선택 옵션: **Teams** (진행도/승인 추가 알림, 긴급 알림).

### 5.1 역할별 진입점

| 역할 | 하는 일 | 진입 방법 |
|------|---------|----------|
| **기획자/PM** | 아이디어 → 구조화 → 카드 생성 | Jira에 아이디어 입력 / 웹 UI / CLI |
| **개발자** | Jira 카드 → 코드 구현 → GitHub PR | Claude Code CLI에서 티켓번호 실행 / Jira 상태 변경 → 자동 트리거 |
| **리뷰어** | CP 승인/반려 | Jira 코멘트 / CLI |
| **관리자** | 프로세스 모니터링 | 대시보드 / `/status` 커맨드 |

### 5.2 커맨드 체계

모든 실행 환경에서 동일한 커맨드 체계를 사용한다. 실행 방식만 다르다.

#### Track A 커맨드

| 커맨드 | 입력 | 출력 | 다음 단계 |
|--------|------|------|----------|
| `idea` | 자유 형식 아이디어 | 구조화된 기획서 | `init-project` 또는 `decompose` |
| `init-project` | 프로젝트 개요 | 전체 분석서, 기술 스택 결정 | `decompose` |
| `decompose` | 분석서 | Epic/Story/Task 분해, 카드 생성 | Track B (`plan`) |

#### Track B 커맨드

| 커맨드 | 입력 | 출력 | 다음 단계 |
|--------|------|------|----------|
| `plan` | 티켓 번호 | 요구사항 분석서 → CP1 | `spec` |
| `spec` | 요구사항 분석서 | 설계 문서 → CP2 | `test` |
| `test` | 설계 문서 | 테스트 케이스/코드 → CP2.5 | `dev` |
| `dev` | 설계 문서 + 테스트 코드 | 구현 코드 | `review` |
| `review` | 구현 코드 | 리뷰 결과 → CP3 | `release` |
| `release` | 리뷰 통과 코드 | 릴리스 노트, 배포 체크리스트 → CP4 | 배포 |

#### 유틸리티 커맨드

| 커맨드 | 용도 |
|--------|------|
| `status` | 현재 티켓의 프로세스 진행 상태 조회 |
| `hotfix` | 프로덕션 긴급 수정 (축약 플로우) |
| `design-proposal` | UI/UX 디자인 제안서 생성 |

### 5.3 알림 및 피드백

| 이벤트 | 알림 대상 | 기본 채널 | 선택: Teams |
|--------|----------|----------|:----------:|
| CP 도달 (승인 필요) | 승인 담당자 | Jira 코멘트 + 알림 | 추가 알림 |
| CP 승인/반려 | 작업 담당자 | Jira 코멘트 + 알림 | 추가 알림 |
| 에스컬레이션 발생 | 시니어/리드 | Jira 긴급 알림 | **긴급 알림** |
| 가드레일 위반 감지 | 보안/인프라 담당자 | Jira 긴급 알림 | **긴급 알림** |
| 작업 완료 (PR 생성) | 개발자 + 리뷰어 | GitHub 알림 | - |
| 배포 완료 | 팀 전체 | Jira 알림 | 추가 알림 |

### 5.4 산출물 조회

- **진행 상태**: `status` 커맨드로 현재 단계, 다음 단계, 블로커 확인
- **산출물 위치**: `artifacts/{ticket_id}/{단계번호}_{단계명}/`
- **리뷰 결과**: `artifacts/{ticket_id}/07_review/` 하위
- **Jira 연동 시**: Jira 티켓 코멘트에 산출물 요약 자동 기록

## 6. 레이어 간 관계

### 동작 예시: `plan` 커맨드 실행

```
Layer 3 (인터페이스)        Layer 2 (실행 엔진)          Layer 1 (규칙)
────────────────          ────────────────           ──────────
/plan MOCA-001      →     오케스트레이터가
                          Jira에서 티켓 조회       ←  Planner 역할 정의
                          Planner 에이전트 실행    ←  산출물 형식
                          산출물 생성/저장              가드레일 체크
                    →     CP1 알림 발송            ←  승인 매트릭스
```

### 교체 가능성

| 교체 대상 | 예시 | 영향 범위 |
|----------|------|----------|
| **Layer 3만 교체** | Claude Code CLI → Teams 봇 | 프로세스·실행은 동일, 진입점만 변경 |
| **Layer 2만 교체** | Claude → GPT 또는 Gemini | 프로세스·인터페이스 동일, AI 엔진만 변경 |
| **Layer 2 일부 추가** | 로컬 CLI + CI/CD 자동 리뷰 병행 | 기존 환경 유지하며 자동화 확장 |
| **Layer 1은 불변** | — | 어떤 도구를 쓰든 규칙은 같다 |

### 의존 방향

```
Layer 3 → Layer 2 → Layer 1
(상위)     (중간)     (하위)

- 상위가 하위를 참조한다 (하위는 상위를 모른다)
- Layer 1 변경 → 모든 레이어에 영향
- Layer 3 변경 → Layer 1, 2에 영향 없음
```

## 7. 시나리오별 동작 흐름

### 7.1 기획자가 중앙 프로젝트에서 아이디어를 내는 경우

```
기획자: [AI-DEV] Jira 프로젝트에 "신규 장치 A 추가" Idea 생성 (Layer 3)
    ↓
실행 엔진: Jira 웹훅 감지 → Claude API로 에이전트 실행 (Layer 2)
    ↓
AI가 Layer 1 규칙에 따라:
    ├─ 관련 GitHub 레포 분석 (server, ios, android, firmware)
    ├─ 기능 분해 (Epic → Story → Task)
    ├─ 플랫폼별 태스크 분리
    ├─ 각 플랫폼 Jira 프로젝트에 카드 자동 생성
    └─ [AI-DEV] 원본에 하위 티켓 링크 연결
    ↓
기획자/PM에게 CP0, CP1 승인 요청 (Layer 3: Jira 알림)
    ↓
승인 후 → 개발자에게 플랫폼별 카드 배정
```

**핵심**: 기획자는 아이디어 텍스트만 입력한다. 플랫폼 분리·카드 생성·링크 연결은 모두 자동.

### 7.2 개발자가 티켓을 실행하는 경우

```
개발자: /plan MOCA-101 (Layer 3: Claude Code CLI)
    또는: Jira에서 카드 상태를 "In Progress"로 변경 (Layer 3: Jira)
    ↓
실행 엔진이 해당 레포에서 에이전트 순차 실행 (Layer 2)
    ↓
Layer 1 규칙에 따라:
    plan → spec → test → dev → review
    각 단계마다 산출물 생성, CP 체크
    ↓
GitHub PR 생성 + 배포 직전 상태
    ↓
사람에게 최종 확인 요청 (Layer 3)
```

**핵심**: 개발자는 티켓 번호만 알면 된다. 나머지는 AI가 자동 실행.

### 7.3 기획자가 플랫폼 프로젝트에 직접 생성하는 경우

```
기획자: [SERVER] Jira 프로젝트에 "장치 A API 추가" Idea 생성
    ↓
Jira 웹훅 → 서버/봇이 감지 (Layer 2)
    ↓
Claude API로 아이디어 구조화 (Layer 1 규칙)
    ├─ 문제 정의, 목표, 대상 사용자
    ├─ 연관 플랫폼 자동 감지 (iOS, Android 등)
    ├─ 질문이 있으면 Jira 코멘트로 질문
    └─ 관련 플랫폼 프로젝트에 연쇄 티켓 생성
    ↓
구조화된 기획서를 Jira 코멘트에 작성
    ↓
기획자 확인 (CP0) → 승인 시 decompose 자동 실행
```

**핵심**: CLI 설치 불필요. 기획자가 아는 플랫폼 프로젝트에 바로 생성하면, AI가 나머지 플랫폼을 알아서 연쇄 처리.
