# AI 산출물 출력 형식

## 목적

AI가 각 단계에서 생성하는 산출물의 표준 출력 형식을 정의한다.
모든 Agent는 이 형식을 따라 결과를 제시하며, 다음 Agent가 입력으로 파싱할 수 있어야 한다.

---

## 공통 규칙

- 모든 산출물은 마크다운 형식으로 작성한다
- 섹션 헤더는 `[섹션명]` 형식을 사용한다
- 사람에게 질문이 필요하면 `[추가 정보 필요]` 섹션에 정리한다
- 마지막에 `[다음 단계]`를 반드시 포함한다
- 산출물은 `artifacts/{ticket_id}/{단계}/` 경로에 저장한다
- 플랫폼별 산출물이 있는 경우 플랫폼 태그 (`[backend]`, `[ios]`, `[android]`, `[web]`)를 명시한다

---

## Traceability (추적성) 규칙

### 목적

요구사항(AC) → 테스트 → 코드 변경 → 리뷰 판정 → Jira 상태가 **양방향으로 추적 가능**해야 한다.
"이 코드가 어떤 요구사항을 충족하는가", "이 AC는 어떤 테스트로 검증되는가"를 즉시 확인할 수 있어야 한다.

### ID 체계

| 단계 | ID 형식 | 예시 |
|------|--------|------|
| 인수 조건 | AC-{번호} | AC-1, AC-2, ... |
| 테스트 케이스 | {카테고리}-{번호} | UT-01, IT-03, ERR-05 |
| 코드 변경 | 커밋 해시 | `8fb87e9` |
| 리뷰 판정 | R{라운드}-{리뷰어} | R1-primary, R2-secondary |

카테고리 약어: UT(단위), IT(통합), UI(UI상태), LC(생명주기), ERR(에러경계값)

### 링크 규칙 — 각 산출물에 필수 포함

#### 1. Test AI → test_cases.md

```
[인수 조건 ↔ 테스트 매핑]
| AC | 테스트 ID | 파일:라인 |
|----|----------|----------|
| AC-1 | UT-01, UI-01 | WidgetProviderTest.kt:45, WidgetUITest.kt:30 |
```

- AC → 테스트 ID 매핑 (기존 규칙)
- **추가**: 테스트 ID → 소스 파일:라인 매핑 (코드에서 역추적 가능하게)

#### 2. Dev AI → change_summary.md

```
[변경 ↔ AC ↔ 테스트 추적]
| 변경 파일:라인 | 관련 AC | 검증 테스트 | 검증 상태 |
|--------------|--------|-----------|----------|
| MocakeyWidgetReceiver.kt:85 | AC-3 | UT-03 | 검증됨 |
| MocaListener.kt:320 | AC-5 | IT-03 | 검증됨 |
| widget_layout.xml:15 | AC-1 | UI-01 | 시각적 확인 필요 |
```

- 코드 변경 → AC → 테스트 3자 매핑 필수
- "테스트 미커버" 항목을 명시 (리뷰어가 별도 확인)

#### 3. Review AI → primary_review.md

```
[AC 충족 추적]
| AC | 코드 근거 (파일:라인) | 테스트 검증 (테스트 ID) | 판정 |
|----|---------------------|---------------------|------|
| AC-1 | widget_layout.xml:10 | UT-01, UI-01 | 충족 |
| AC-5 | MocaListener.kt:320 | IT-03 | 충족 |
```

- AC → 코드 근거 → 테스트 검증까지 **3단 추적** 필수
- 코드 근거 없이 "충족"이라고 적으면 반려

#### 4. 오케스트레이터 → process_log.md 또는 Jira 코멘트

```
[실행 증빙]
| 단계 | 산출물 | 커밋 | 판정 |
|------|--------|------|------|
| 테스트 | test_cases_android.md | - | CP2.5 통과 |
| 구현 | change_summary_android.md | 8fb87e9 | 게이트 통과 |
| 리뷰 R1 | primary_review.md | - | 반려 |
| 구현 수정 | change_summary_android.md (v2) | 3d0299f | 게이트 통과 |
| 리뷰 R2 | primary_review_r2.md | - | 승인 |
```

### 역추적 예시

누군가 "AC-5는 어떻게 구현되었나?"라고 물으면:

1. `test_cases.md` → AC-5 → 테스트 ID: UT-04, IT-03
2. `change_summary.md` → AC-5 → MocaListener.kt:320 (uploadWidgetEventLog)
3. `primary_review.md` → AC-5 → 충족 (코드 근거: MocaListener.kt:320)
4. Git → `3d0299f` (반려 수정 커밋에서 추가)

## Spec AI 출력 형식

파일: `artifacts/{ticket_id}/02_spec/design_doc.md`

```
[설계 개요]
- 변경 목적 1줄 요약
- 관련 요구사항 참조 (01_planner 링크)

### [backend] 신규 엔드포인트

POST /api/v1/cards/issue
- 설명: 카드 발급 요청
- 요청:
  {
    "userId": "string",
    "siteId": "string",
    "type": "common | personal"
  }

  {
    "cardId": "string",
    "status": "pending",
    "issuedAt": "ISO8601"
  }

### [backend] 수정 엔드포인트
- 기존 스펙과 변경 사항 diff

[DB 변경]

### 스키마 변경

- Prisma 모델 정의

- 마이그레이션 SQL 또는 Prisma 마이그레이션 명령

[플랫폼별 영향]

### [ios]
- 연동할 API 목록
- 로컬 데이터 모델 변경 사항

### [android]
- 연동할 API 목록
- 로컬 데이터 모델 변경 사항

### [web]
- 연동할 API 목록

[다음 단계]

## 프로토타입 출력 형식

Spec AI 이후, Designer AI 이전에 동작 확인이 필요한 경우 생성한다.
프로토타입은 **화면 구현 + 핵심 기능 최소화**로, 빌드해서 기기/시뮬레이터에서 직접 돌려보는 것이 목적이다.

파일: `artifacts/{ticket_id}/02.5_prototype/prototype_summary.md`

```
[프로토타입 개요]
- 목적: 이 프로토타입으로 무엇을 확인하는가
- 브랜치: feature/{ticket_id}-prototype

[포함 범위]
- 구현한 화면 목록
- 포함한 핵심 기능 (최소 동작)
- 사용한 더미 데이터

[미포함 범위]
- 생략한 것 (에러 처리, 테스트, 실제 API 연동 등)

[빌드 및 실행 방법]
- 플랫폼별 빌드 명령어
- 필요한 사전 조건 (로그인 상태, 테스트 데이터 등)
- 확인 방법 (시뮬레이터에서 이렇게 하면 동작을 볼 수 있다)

[확인 포인트]
- [ ] 확인 항목 1 (예: "위젯이 홈 화면에 정상 표시되는가")
- [ ] 확인 항목 2 (예: "문열기 버튼 탭 시 앱이 실행되는가")

[피드백 요청]
- "이 화면/동작이 원하는 방향인가?"
- "빠진 기능이나 잘못된 흐름이 있는가?"

[다음 단계]
- 승인 시: 프로토 브랜치 삭제 → 본 브랜치에서 Designer AI (와이어프레임)로 전달
- 반려 시: 피드백 반영하여 재작업 (최대 3회)
```

## Test AI 출력 형식

파일: `artifacts/{ticket_id}/04_test/test_cases.md`

**주의: 인수 조건 매핑표가 없는 산출물은 불완전하다. 반드시 첫 번째 섹션으로 작성한다.**

```
[인수 조건 ↔ 테스트 매핑]

| # | 인수 조건 | 테스트 ID | 카테고리 | 검증 방법 |
|---|----------|----------|---------|----------|
| AC-1 | 위젯이 3가지 상태를 정확히 표시한다 | T-1, T-2, T-3 | UI 상태 | 단위 테스트 |
| AC-2 | 문열기 버튼 탭 시 BLE 스캔 수행 | T-10, T-11 | 생명주기 | 단위+통합 테스트 |
| AC-3 | 카드 변경 시 위젯 즉시 갱신 | T-20, T-21 | 통합·트리거 | 단위 테스트 |
| ... | ... | ... | ... | ... |

**매핑 안 된 인수 조건: 없음** ← 이 줄이 반드시 있어야 함. 있으면 테스트 추가 후 재매핑.

[테스트 케이스]

### 카테고리 1: 단위 테스트
| # | 시나리오 | 입력 | 기대 결과 | assert 대상 |
|---|---------|------|----------|-----------|
| T-1 | countActiveCards — activated만 카운트 | [activated, suspended] | 1 반환 | 반환값 == 1 |

### 카테고리 2: 통합·트리거 테스트
| # | 트리거 이벤트 | 기대 연쇄 동작 | 검증 방법 |
|---|-------------|-------------|----------|
| T-20 | 카드 추가 (write 연산) | WidgetDataManager 갱신 + 위젯 리로드 호출 | 메서드 호출 검증 (mock/spy) |

### 카테고리 3: UI 상태 테스트
| # | 입력 상태 | 기대 UI | 검증 방법 |
|---|----------|--------|----------|
| T-30 | isLoggedIn=false | "로그인 필요" 텍스트 표시, 문열기 버튼 숨김 | 텍스트 값 + visibility assert |

### 카테고리 4: 생명주기 테스트
| # | 컴포넌트 | 시나리오 | 기대 동작 |
|---|---------|---------|----------|
| T-40 | WidgetBLEService | onStartCommand 호출 | START_NOT_STICKY 반환 + startForeground 호출 |

### 카테고리 5: 에러·경계값 테스트
| # | 시나리오 | 입력 | 기대 결과 |
|---|---------|------|----------|
| T-50 | 빈 App Group에서 restore | App Group 비어있음 | 크래시 없이 정상 완료 |

[수정 검증 테스트]
AI가 코드를 수정한 후 "변경이 실제로 반영되었는지" 확인할 수 있는 테스트 목록:
- T-20: 카드 추가 → 위젯 갱신 트리거 호출 여부
- T-30: 상태별 UI 텍스트/visibility가 실제로 올바른 값인지
- ...

[테스트 코드 파일 목록]
- ios: artifacts/{ticket_id}/04_test/ios/WidgetTests.swift
- android: artifacts/{ticket_id}/04_test/android/WidgetDataManagerTest.kt

[테스트 실행 확인]
**테스트 작성 후 반드시 실행하여 결과를 기재한다. 실행 0개 = 미완성.**

| 플랫폼 | 실행 수 | 통과 | 실패 | 비고 |
|--------|--------|------|------|------|
