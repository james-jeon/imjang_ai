# 27. 지식 관리 & 교훈 기록 (Knowledge Management)

> 프로젝트에서 발생한 이슈, 해결 방법, 교훈을 체계적으로 축적하고 재활용하는 프로세스.

---

## 1. 목적

1. **같은 실수 반복 방지** — 한 번 해결한 문제의 원인과 해법을 기록
2. **AI 에이전트 학습 자료** — CLAUDE.md와 프로세스 문서에 교훈 반영
3. **온보딩 가속** — 새 팀원이 과거 이슈를 빠르게 파악
4. **프로세스 개선** — 반복 이슈가 프로세스 결함을 드러내면 문서 업데이트

---

## 2. 교훈 기록 구조

### 2.1 Lessons Learned 항목

```markdown
## [LL-XXX] 제목

- **날짜**: 2024-01-15
- **프로젝트**: WIDGET-001
- **카테고리**: 테스트 / 빌드 / 배포 / 설계 / 보안 / 성능
- **심각도**: Critical / High / Medium / Low

### 문제
(무엇이 발생했는가)

### 원인
(왜 발생했는가 — Root Cause)

### 해결
(어떻게 해결했는가)

### 교훈
(앞으로 어떻게 방지할 것인가)

### 반영된 문서
(어떤 프로세스 문서가 업데이트되었는가)
```

### 2.2 실제 예시

```markdown
## [LL-001] iOS 테스트 0개 실행 — pbxproj 미등록

- **날짜**: 2024-12-20
- **프로젝트**: WIDGET-001
- **카테고리**: 테스트
- **심각도**: Critical

### 문제
Test AI가 테스트 파일을 생성했으나, 실제 실행 시 0개 테스트가 실행됨.
빌드는 성공하여 문제를 발견하기 어려웠음.

### 원인
Swift 테스트 파일이 디스크에 존재하지만 `.pbxproj`의 4개 섹션
(PBXFileReference, PBXBuildFile, PBXGroup, PBXSourcesBuildPhase)에
등록되지 않아 Xcode가 인식하지 못함.

### 해결
1. `.pbxproj`에 4개 섹션 수동 등록
2. `@testable import` 대상을 올바른 모듈로 변경

### 교훈
- "빌드 성공 ≠ 테스트 실행" — 테스트 개수를 반드시 확인
- Test AI 산출물에 "테스트 실행 확인" 섹션 추가
- CP2.5 체크리스트에 "실행 0개 = 즉시 반려" 규칙 도입

### 반영된 문서
- `03_ai_dev_process.md` — CP2.5, CP3 체크리스트
- `13_output_format.md` — 테스트 실행 확인 섹션
- `15_agent_prompts_guide.md` — Test AI 플랫폼별 함정
```

## 3. 기록 시점과 주체

| 시점 | 주체 | 기록 대상 |
|------|------|----------|
| CP2.5 리뷰 반려 시 | 리뷰어 | 반려 사유가 프로세스 결함이면 |
| CP3 리뷰 반려 시 | 리뷰어 | 반복 패턴이면 |
| 프로덕션 장애 발생 시 | 장애 대응자 | Post-mortem으로 |
| 배포 실패 시 | 배포 담당자 | CI/CD 관련 이슈 |
| 스프린트 회고 시 | 팀 전체 | 프로세스 개선점 |

## 4. 교훈 → 프로세스 반영 플로우

```
이슈 발생
  ↓
교훈 기록 (LL-XXX)
  ↓
카테고리 분류
  ↓
해당 프로세스 문서 식별
  ↓
문서 업데이트 (규칙 추가/변경)
  ↓
CLAUDE.md에 반영 (AI 에이전트가 즉시 적용)
  ↓
팀 공유 (Slack #dev)
```

```
    └── ADR-002-...
```

### 5.1 ADR (Architecture Decision Record)

중요 기술 결정은 ADR로 기록:

```markdown
## ADR-XXX: 제목

- **상태**: 제안 / 승인 / 폐기
- **날짜**: 2024-01-15
- **결정자**: 이름

### 맥락
(왜 이 결정이 필요했는가)

### 결정
(무엇을 선택했는가)

### 대안
(어떤 대안이 있었고 왜 선택하지 않았는가)

### 결과
(이 결정의 영향은 무엇인가)
```

## 6. 검색 & 활용

### 6.1 태그 시스템

교훈에 아래 태그 부착하여 검색 용이하게:

```
#ios #android #test #build #deploy #security #performance
#pbxproj #gradle #cocoapods #widget #ble #api
```

### 6.2 AI 에이전트 활용

- **Dev AI**: 구현 시작 전 해당 모듈의 과거 교훈 검색
- **Test AI**: 테스트 설계 시 과거 에러 패턴 참조
- **Review AI**: 리뷰 시 과거 반려 사유 패턴 확인

---

## 7. AI 에이전트 규칙

1. **새 이슈 발견 시** → 교훈 기록 여부 제안. 단, 기록 자체는 인간이 승인
2. **구현 전** → 관련 모듈의 기존 교훈(LL-XXX) 확인 후 반영
3. **리뷰 반려 시** → 반려 사유가 기존 교훈과 일치하면 해당 LL 참조
4. **프로세스 문서 수정 시** → 수정 근거가 되는 LL 번호 명시
5. **ADR 작성** → AI가 초안 작성, 인간이 승인. 결정은 항상 인간이 내림
