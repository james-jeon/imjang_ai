# AI 기획 단계

## 목적

프로젝트 시작 또는 기능 추가 시, AI가 기획 과정을 보조하여
기획서 품질을 높이고 Jira 초안 작성 비용을 줄인다.

프로세스 진입점에 따라 두 가지 흐름이 있다. (03_ai_dev_process 참조)

---

## 흐름 A: 프로젝트 시작 (신규/리뉴얼)

전체 서비스를 한번에 분석하고 Jira 구조를 일괄 생성한다.

```
프로젝트 정의 (사람)
    ↓
AI 전체 분석 (레거시 분석 또는 요구사항 구조화)
    ↓
기술 스택 + 시스템 설계 (AI 자율 판단)
    ↓
CP0: 프로젝트 방향 승인 (사람)
    ↓
기능 분해 → Epic / Story / Task 일괄 생성
    ↓
Jira 구조 일괄 생성 + 우선순위 정렬
    ↓
CP1: 전체 범위 확인 (사람)
    ↓
Sprint 단위로 흐름 B 반복 실행
```

상세 단계는 03_ai_dev_process의 Track A를 참조한다.

---

## 흐름 B: 기능 추가 (운영 중)

개별 아이디어에서 Jira 단건을 생성한다.

```
아이디어 (사람)
    ↓
AI 기획 보조
    ↓
기획서 초안 (AI 생성 → 사람 검토)
    ↓
Jira 티켓 자동 생성
    ↓
이후 프로세스 (03_ai_dev_process Track B)
```

---

## 1단계: 아이디어 입력

사람이 아이디어를 자유 형식으로 전달한다.

형태는 자유롭다:
- 한 줄 메모: "출입 인증 실패할 때 사용자가 뭘 해야 하는지 모름"
- 음성 메모 텍스트 변환
- CS 문의 데이터 요약
- 경쟁 서비스 스크린샷 + 코멘트

AI가 처리 가능한 최소 입력:
- **무엇이 문제인가** 또는 **무엇을 만들고 싶은가** (1문장 이상)

---

## 2단계: AI 기획 보조

AI가 아이디어를 분석하여 다음을 수행한다.

### 2.1 아이디어 구조화

자유 형식 입력을 다음 구조로 정리한다:

- 문제 정의: 현재 무엇이 문제인가
- 목표: 이 기능/개선이 달성하려는 것
- 대상 사용자: 누구를 위한 것인가
- 기대 효과: 어떤 변화가 생기는가

### 2.2 기존 시스템 분석

코드베이스와 기존 문서를 분석하여:

- 관련된 기존 기능이 있는지 확인
- 영향받는 시스템 범위 사전 파악 (앱/웹/백엔드)
- 유사하게 구현된 패턴이 있는지 조사
- 기술적 제약 사항 사전 파악

### 2.3 실현 가능성 검토

- 기존 아키텍처에서 구현 가능한 방식 제안
- 예상되는 기술적 난이도 (low / medium / high)
- 필요한 외부 의존성 (새로운 라이브러리, API 등)
- 예상 작업 규모 (소/중/대)

### 2.4 추가 질문

아이디어가 모호하면 AI가 기획 방향을 잡기 위한 질문을 한다:

```
[기획 질문]

1. 이 기능은 모든 사용자 대상인가요, 특정 권한의 사용자만 대상인가요?
2. 기존 "설정" 화면에 추가하는 것인가요, 별도 화면이 필요한가요?
3. 이 문제의 발생 빈도는 어느 정도인가요? (일간 CS 건수 등)
```

---

## 3단계: 기획서 초안 생성

AI가 구조화된 기획서 초안을 생성한다.

### 기획서 형식

```
[기획서]

제목: (기능/개선 이름)

1. 배경
   - 현재 상황
   - 문제점

2. 목표
   - 이 기획이 달성하려는 것

3. 대상 사용자
   - 누구를 위한 것인가

4. 기능 요구사항
   - 주요 기능 나열

5. 기존 시스템 영향
   - 관련 기존 기능
   - 영향받는 시스템 (앱/웹/백엔드)

6. 기술적 검토
   - 구현 방향 제안
   - 예상 난이도
   - 제약 사항

7. 범위 외 (Out of Scope)
   - 이번에 다루지 않는 것

8. 예상 Jira 티켓
   - 이 기획에서 생성될 Jira 티켓 목록 (초안)
```

### 사람 검토

- 사람이 기획서를 검토하고 방향을 확정한다
- 수정이 필요하면 코멘트를 달고 AI가 반영한다
- **이 시점이 가장 중요한 의사결정 포인트** — 방향이 틀리면 이후 전체가 틀어진다

---

## 4단계: Jira 티켓 자동 생성

기획서가 승인되면 AI가 Jira 티켓을 자동 생성한다.

- 기획서의 "예상 Jira 티켓" 섹션을 기반으로 생성
- 각 티켓은 06_jira_template의 필수 항목 + 메타 필드를 자동으로 채움
- 티켓 간 의존성도 자동으로 설정
- 생성된 티켓은 사람이 최종 확인 후 확정

---

## 프로젝트 초기 설정 (신규/리뉴얼 프로젝트 시)

신규 프로젝트 또는 레거시 리뉴얼 시, Planner AI가 기획 단계에서 프로젝트 기반을 자율적으로 판단하고 설정한다.

### AI 자율 판단 영역

다음 항목들은 AI가 분석 결과를 기반으로 직접 결정한다. 사람은 CP2에서 전체 방향을 승인/수정한다.

| 항목 | AI가 하는 일 | 판단 기준 |
|------|------------|----------|
| 기술 스택 | 프로젝트 요구사항과 기술 생태계를 분석하여 선정 | AI 코드 생성 효율, 생태계 성숙도, 의존성 최소화, 타입 안전성 |
| 아키텍처 패턴 | 프로젝트 규모와 특성에 맞는 아키텍처 결정 | 서비스 규모, 팀 규모, 배포 전략 |
| 프로젝트 구조 | 디렉토리 구조, 모노레포/멀티레포 결정 | 플랫폼 간 코드 공유 필요성, CI/CD 효율 |
| 디자인 시스템 | 기존 앱/서비스 분석하여 디자인 토큰 추출 | 기존 비주얼 일관성, 플랫폼 가이드라인 |
| SDK/라이브러리 인터페이스 | 기존 코드를 분석하여 인터페이스 스펙 자동 생성 | 기존 호출 패턴, 변경 최소화 |
| API 설계 | 기존 API를 분석하여 새 스펙 생성 | RESTful 규칙, 리소스 정리/통합 |

### 기술 스택 선정 기준

AI는 다음 기준을 우선순위 순으로 적용한다:

1. **AI 코드 생성 효율** — 해당 기술의 학습 데이터가 풍부하고 AI가 정확한 코드를 생성할 수 있는가
2. **선언형 패러다임 우선** — 선언형 UI/코드는 AI가 생성하기 쉽고 코드량이 적다
3. **네이티브/표준 솔루션 우선** — 서드파티 의존성보다 플랫폼 네이티브 솔루션을 우선한다
4. **타입 안전성** — 정적 타입 언어/프레임워크를 우선한다
5. **플랫폼 간 일관성** — 가능하면 유사한 패턴/패러다임을 플랫폼 간 통일한다
6. **생태계 성숙도** — 충분한 커뮤니티와 문서가 있는 기술을 선택한다
7. **마이그레이션 비용** — 기존 스택과의 전환 비용을 고려한다

### 레거시 리뉴얼 시 추가 분석

레거시 코드베이스가 있을 경우 AI가 자동으로 수행하는 분석:

```
레거시 코드 입력
    ↓
1. 프로젝트 규모 측정 (파일 수, 코드 라인, 엔드포인트/화면 수)
2. 기존 기술 스택 식별 (프레임워크, 언어, 라이브러리)
3. 핵심 기능 추출 (도메인 모듈, API, 화면 목록)
4. 외부 연동 식별 (DB, 외부 API, SDK)
5. 재사용 가능 요소 판별 (유지할 SDK, 변경할 인터페이스)
    ↓
기술 스택 결정 + 리뉴얼 범위 정의 + 실행 계획 수립
    ↓
CP2: 사람 승인
```

## AI가 기획 단계에서 하지 않는 것

- 비즈니스 우선순위 결정 (사람의 판단)
- 일정 확정 (사람의 판단)
- UX/UI 디자인 확정 (Designer AI가 별도 단계에서 수행)
- 예산 관련 의사결정
