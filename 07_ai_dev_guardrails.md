# AI 개발 가드레일

## 목적

AI의 자동 변경으로 인한 안전 문제를 방지한다.

## 제한 영역

AI가 단독으로 변경할 수 없는 영역:

- 데이터베이스 스키마 마이그레이션
- 결제 시스템
- 인증 시스템
- 보안 설정
- 인프라 네트워킹

## 제한 영역 탐지 방법

AI가 제한 영역을 자동으로 감지하기 위한 규칙:

### 파일 경로 패턴

| 제한 영역 | 경로 패턴 예시 |
|-----------|---------------|
| DB 마이그레이션 | `**/migrations/**`, `**/schema/**` |
| 결제 시스템 | `**/payment/**`, `**/billing/**` |
| 인증 시스템 | `**/auth/**`, `**/authentication/**`, `**/security/**` |
| 보안 설정 | `**/config/security/**`, `**/*.key`, `**/*.pem` |
| 인프라 | `**/infra/**`, `**/terraform/**`, `**/k8s/**` |

### 모듈 태그

- 프로젝트 내 모듈이나 패키지에 `@guardrail` 태그를 부여하여 명시적으로 제한 영역을 표시한다
- AI는 변경 대상 파일이 태그된 모듈에 속하면 자동으로 사람 승인을 요청한다

## 승인 워크플로

제한 영역에 대한 변경이 필요한 경우:

```
AI가 제한 영역 변경 감지
  ↓
자동으로 작업 중단
  ↓
Jira 티켓에 guardrail-review 라벨 부착
  ↓
지정된 승인자에게 알림 (Slack / 이메일)
  ↓
승인자가 변경 내용 검토
  ↓
승인 → AI가 작업 재개 / 거부 → AI가 대안 제시 또는 에스컬레이션
```

### 승인 기록

- 모든 가드레일 승인/거부는 Jira 티켓에 로그로 남긴다
- 승인자, 승인 시각, 대상 파일, 변경 내용 요약을 기록한다

## SDK 변경 정책

### SDK는 범용 라이브러리다

MocaBLESDK 등 SDK는 특정 앱 전용이 아니라 **여러 앱에서 공통으로 사용**하는 범용 라이브러리다. 앱 기능 구현 과정에서 SDK 내부 수정이 필요할 수 있지만, 변경의 영향 범위가 앱 단독 변경보다 넓다.

### SDK 수정이 필요한 경우

앱 기능 구현 시 SDK의 저장소 접근 방식, 프로세스 간 데이터 공유, 플랫폼 Extension 지원 등 SDK 내부 로직 수정이 불가피한 경우가 있다. 이때 다음을 따른다:

| 항목 | 규칙 |
|------|------|
| PR 분리 | SDK 변경은 앱 변경과 **별도 PR**로 분리한다 |
| 영향도 분석 | SDK를 사용하는 **모든 앱**에 대한 영향도를 분석하고 문서화한다 |
| 하위 호환성 | 기존 앱이 수정 없이 동작해야 한다. 불가능하면 마이그레이션 가이드를 제공한다 |
| 버전 관리 | SDK 변경 시 버전을 올리고, 변경 내역(CHANGELOG)을 기록한다 |
| 테스트 범위 | SDK를 사용하는 앱별로 빌드 검증을 수행한다 |

### 앱 전용 변경 vs SDK 변경 구분 기준

| 구분 | 앱 전용 변경 | SDK 변경 |
|------|------------|---------|
| 범위 | 해당 앱에만 영향 | SDK를 사용하는 모든 앱에 영향 |
| 예시 | 위젯 UI, 상태 분기, 데이터 갱신 로직 | UserDefaults 저장소 경로, Keychain 접근 방식, DB 경로 |
| PR | 앱 PR에 포함 | 별도 SDK PR |
| 리뷰 | 일반 CP3 리뷰 | CP3 리뷰 + 영향도 분석 필수 |

- 프로덕션 배포
