# AI Agent 아키텍처

## 개요

AI 시스템은 역할별 전문 Agent로 구성한다.
각 Agent는 명확한 입력과 출력을 가지며, 오케스트레이터가 Agent 간 데이터 전달을 관리한다.
Agent는 플랫폼(앱/웹/백엔드)에 따라 플랫폼 전용 동작을 수행한다.

## 2개의 Phase

전체 프로세스는 **기획 Phase**와 **개발 Phase**로 나뉜다.

### 기획 Phase (`/idea`)
아이디어를 받아 분석 + 기획 + 설계까지 완료한 후, 상세한 Jira 카드를 생성한다.
Ideator → Planner → Spec 역할을 순차 수행하며, 기획 리뷰(CP) 후 Jira를 생성한다.

### 개발 Phase (`/in-progress`)
Jira 스토리 하나를 받아 테스트 → 구현 → 리뷰 → 릴리즈까지 순차 실행한다.
Test → Dev → Review → Release 역할을 순차 수행한다.

## Agent 구성

### Ideator AI
- 역할: 자유 형식 아이디어를 분석하고, 대상 레포를 결정하며, 코드베이스를 분석한다
- 입력: 자유 형식 아이디어 텍스트
- 출력: 아이디어 분석 결과, 대상 레포 매핑, 코드베이스 분석 결과
- 수행 내용:
  - CLAUDE.md 레포 레지스트리에서 역할/주요 기능 기반으로 대상 레포 자동 매핑
  - GitHub MCP 또는 로컬 코드로 대상 레포의 아키텍처, 관련 모듈 분석
  - 기존 유사 기능 존재 여부 확인
  - 실현 가능성 검토 (난이도, 외부 의존성, 예상 규모)
  - 모호한 부분은 사람에게 질문

### Planner AI
- 역할: 요구사항을 확장하고 작업을 분해한다
- 입력: Ideator AI의 분석 결과, 또는 기존 Jira 티켓
- 출력: 확장된 요구사항, 인수 조건, 작업 분해, 의존성 그래프
- 수행 내용:
  - 요구사항 구체화 (검증 가능한 인수 조건)
  - 플랫폼별 스토리로 작업 분해
  - 스토리 간 의존성 설정
  - risk_level 판정 (06_jira_template.md 기준)
- 플랫폼별 동작:
  - 멀티 플랫폼 작업 시 플랫폼별 태스크를 분리하여 의존성 그래프에 반영한다
  - 예: "백엔드 API 먼저 → 앱 구현 병렬"

### Spec AI
- 역할: 설계 방향을 수립하고, API/DB 변경을 정의한다
- 입력: Planner AI의 작업 분해 결과 + Ideator AI의 코드 분석 결과
- 출력: 설계 방향, API 스펙, DB 스키마 변경 정의, 기술 제약사항
- 수행 내용:
  - 각 스토리별 사용할 클래스/모듈, 아키텍처 패턴 결정
  - API 변경 시 스펙 초안
  - DB 변경 시 스키마 변경 초안
  - 플랫폼별 기술 제약사항 정리
- 플랫폼별 산출물:
  - 백엔드: API 스펙 (OpenAPI), DB 스키마, 아키텍처 결정 기록 (ADR), 부하 예측, 관찰성 설정
  - 앱: 화면별 API 연동 스펙, 로컬 저장소 스키마, 네이티브 모듈 연계 정의
  - 웹: 페이지/컴포넌트 구조, 상태 관리 설계, SEO 요구사항

### Designer AI
- 역할: UI/UX 설계를 수행한다 (requires_ui_change == yes일 때 실행)
- 2단계로 동작한다

#### 1차 디자인 (구현 전 — 와이어프레임)
- 시점: 기획 Phase에서 Spec 후, 또는 개발 Phase에서 Test 전
- 입력: Spec AI의 설계 문서 + 기존 디자인 시스템/컴포넌트
- 출력: UI/UX 플로우, 와이어프레임, 화면 구조, 인터랙션 정의
- 목적: 구조와 플로우를 확정하여 Dev AI가 올바른 방향으로 구현하도록 한다
- 사람 검토: 새로운 UX 패턴이면 사람 확인 필수, 기존 패턴 재사용이면 자동 진행
- 플랫폼별 고려:
  - 앱: iOS Human Interface Guidelines / Android Material Design 준수
  - 웹: 반응형 레이아웃, 크로스브라우저 호환성

#### 2차 디자인 (구현 후 — 비주얼 적용)
- 시점: 기능 구현 완료 후, 최종 리뷰 전
- 입력: Dev AI의 구현 결과 + 1차 와이어프레임 + 디자인 시스템
- 출력: 비주얼 디자인 스펙 (색상, 타이포그래피, 간격, 아이콘, 애니메이션) + 적용 코드
- 목적: 구현된 기능에 최종 비주얼 디자인을 입힌다
- 사람 검토: 비주얼 결과물을 사람이 확인한다
- 플랫폼별 고려:
  - 앱: 다크 모드 대응, 해상도별 에셋 생성, 플랫폼 가이드라인 준수
  - 웹: 접근성 색상 대비 (WCAG AA 4.5:1 이상), 반응형 브레이크포인트별 비주얼 확인, 다크 모드 대응

### Test AI
- 역할: 인수 조건을 **빠짐없이** 테스트로 변환한다. "자기 판단으로 충분하다"고 넘어가지 않는다
- 입력: Jira 스토리의 인수 조건 + 설계 방향
- 출력: 인수 조건 ↔ 테스트 매핑표 + 테스트 코드 (5개 카테고리: 단위 / 통합·트리거 / UI 상태 / 생명주기 / 에러·경계값)
- 핵심 원칙:
  - **인수 조건 전수 매핑**: 매핑되지 않는 인수 조건이 있으면 테스트를 추가한다
  - **수정 검증 가능**: AI가 코드 수정 시 변경이 반영되었는지 테스트로 직접 확인할 수 있어야 한다
  - **결과값 assert**: View 인스턴스 생성만 확인하는 테스트는 불완전. 텍스트, visibility, 호출 횟수 등 구체적 값을 검증한다
  - **자기 검증 필수**: 제출 전 자기 검증 체크리스트를 전부 통과해야 한다
- 플랫폼별 산출물:
  - 백엔드: 단위 테스트, 통합 테스트, API 계약 테스트, 부하 테스트 시나리오
  - 앱: 단위 테스트, 통합·트리거 테스트, UI 상태 테스트, 생명주기 테스트, 디바이스별 호환성 테스트 매트릭스
  - 웹: 단위 테스트, E2E 테스트, 크로스브라우저 테스트 매트릭스, 접근성 테스트

### Test Validator AI
- 역할: Test AI 산출물의 구조적 품질을 **자동 검증**한다. 교차 리뷰 전에 실행하여 명백한 품질 미달을 사전 차단한다
- 입력: Test AI 산출물 (test_cases.md + 테스트 코드) + 인수 조건 원본 + 실제 소스 코드
- 출력: `artifacts/{ticket_id}/04_test/validation_result.md` (V-1~V-12 체크리스트 결과 + 판정)
- 검증 항목 (12개):
  - 구조: AC 매핑 완전성, 5개 카테고리, 매핑표 존재, Traceability 링크
  - 코드 품질: Placeholder 금지(XCTFail/TODO), Assert 존재, Class.forName 전용 금지
  - API 정합성: Mock API 일치, 상수 존재, import 유효성
  - 플랫폼: 양 플랫폼 AC 커버, 테스트 수준 균형
- 판정: 통과 → 교차 리뷰 진행 / 반려 → Test AI 재실행
- 상세: `.claude/agents/test-validator.md`

### Dev AI
- 역할: 코드를 구현하거나 수정한다
- 입력: Jira 스토리의 설계 방향 + Test AI의 테스트 코드 + Designer AI의 디자인 스펙 (UI 변경 시)
- 출력: 소스 코드 변경 (출력 형식은 13_output_format 참조)
- 변경 제시 시 적용 위치(파일/함수)와 예상 부작용을 함께 명시한다
- 플랫폼별 동작:
  - 백엔드: 서버 코드, 마이그레이션 스크립트, 인프라 설정 (IaC)
  - 앱: iOS (Swift/Kotlin) 네이티브 코드, 플랫폼별 빌드 설정
  - 웹: 프론트엔드 코드, 빌드 설정, 환경별 설정

### Review AI
- 역할: 코드 품질과 요구사항 충족 여부를 검증한다
- 입력: Dev AI의 코드 변경 + Jira 스토리의 인수 조건
- 출력: 리뷰 코멘트, 승인/반려 판정
- 플랫폼별 검증 항목:
  - 백엔드: API 보안, 쿼리 성능, 에러 핸들링, 관찰성 설정 누락 확인
  - 앱: 메모리 누수, 배터리 소모, 오프라인 동작, 플랫폼 가이드라인 준수
  - 웹: 접근성 (WCAG), 성능 (Core Web Vitals), SEO, 크로스브라우저 호환성

### Release AI
- 역할: 릴리즈 준비와 배포 지침을 생성한다
- 입력: 승인된 PR + Jira 스토리의 요구사항
- 출력: 플랫폼별 릴리즈 산출물
- 플랫폼별 산출물:
  - 백엔드:
    - 릴리즈 노트
    - 마이그레이션 실행 순서
    - 환경변수 변경 사항
    - 모니터링 대시보드/알림 설정
    - 롤백 절차
  - 앱:
    - 릴리즈 노트 (사용자용 + 내부용)
    - 스토어 메타데이터 (설명, 변경사항)
    - 스토어 심사 체크리스트
    - 인증서/프로비저닝 프로파일 확인
    - 단계적 출시 (Phased Release) 계획
  - 웹:
    - 릴리즈 노트
    - CDN 캐시 무효화 대상
    - 배포 URL 확인
    - 환경별 설정 변경 사항

## Agent 간 데이터 흐름

```
기획 Phase (/idea)
═══════════════════════════════════════════════════════
아이디어 (자유 형식 텍스트)
    ↓
Ideator AI → 아이디어 분석, 대상 레포 매핑, 코드 분석
    ↓
Planner AI → 요구사항 확장, 인수 조건, 작업 분해
    ↓
Spec AI → 설계 방향, API/DB 스펙, 기술 제약사항
    ↓
교차 리뷰 → 프로토타입 필요성 자동 판정
    ↓
프로토타입 (리스크 기반)
    ├── 기술 리스크 → 기술 검증 PoC (미확인 항목 최소 코드 검증)
    ├── UX 리스크 → 디자인 시안 (HTML 2~3안)
    └── 없음 → 스킵
    ↓
[기획 리뷰 CP] → 사람 승인
    ↓
Jira 생성 (에픽 + 스토리, 상세한 상태)


개발 Phase (/in-progress)
═══════════════════════════════════════════════════════
Jira 스토리 (상세 기획 + 설계 포함)
    ↓
Designer AI [1차] → 와이어프레임 (requires_ui_change == yes)
    ↓
Test AI → 테스트 케이스, 테스트 코드
    ↓
Test Validator AI → 자동 검증 (V-1~V-12) → 반려 시 Test AI 재실행
    ↓
Dev AI → 소스 코드 변경 (기능 구현)
    ↓
Designer AI [2차] → 비주얼 디자인 적용 (requires_ui_change == yes)
    ↓
Review AI → 리뷰 결과 (승인/반려)
    ↓ (승인 시)
Release AI → 릴리즈 산출물
```

## 예외 처리 흐름

### Review AI가 반려한 경우

```
Review AI 반려 → 반려 사유를 Dev AI에 전달 → Dev AI 재구현 → Review AI 재리뷰
```

최대 3회 반복 후에도 통과하지 못하면 에스컬레이션한다. (09_escalation_policy 참조)

### Agent가 입력을 해석할 수 없는 경우

어떤 Agent든 입력이 불충분하거나 모호하면 작업을 중단하고 에스컬레이션한다.
오케스트레이터가 이를 감지하여 사람에게 알린다.

## 오케스트레이터

중앙 오케스트레이터가 Agent 간 작업 흐름을 관리한다.

### 역할

- Agent 실행 순서를 제어한다
- Agent 간 산출물 전달을 중개한다
- 플랫폼별 태스크를 병렬/순차로 스케줄링한다 (예: 백엔드 API 완료 후 앱/웹 구현 병렬 진행)
- Human Checkpoint 시점에 사람에게 알림을 보낸다
- Agent 실패 시 에스컬레이션을 트리거한다

### 설계 방식

오케스트레이터는 **상태 머신(워크플로 엔진)**으로 구현한다. Agent 간 흐름이 확정적이므로 LLM이 흐름을 판단할 필요가 없다.

```
오케스트레이터 = 상태 머신 + DB (워크플로 상태 관리)
각 단계 = 실행 엔진을 통한 Agent 호출 (도구 상세는 14_tool_integration 참조)
```

### 워크플로 상태 관리

각 Jira 티켓 = 하나의 워크플로 인스턴스로 관리한다.

```
[워크플로 테이블]
ticket_id | current_stage | status | platform | risk_level | artifacts_path | retry_count

상태값: pending → running → waiting_approval → approved → running → ... → completed
```

### 단계 전이 규칙

기획 Phase:
```
ideator → planner → spec → [기획 리뷰 CP] → jira_create
```

개발 Phase:
```
designer_1(조건부) → test → dev → designer_2(조건부) → review → release
```

각 단계에서:
1. 현재 단계의 Agent를 호출 (실행 엔진 + 시스템 프롬프트 + 이전 산출물)
2. 결과를 artifacts에 저장
3. CP 지점이면 status를 waiting_approval로 변경, Jira/Slack 알림
4. 승인 수신 시 다음 단계로 전이
5. 실패 시 retry_count 증가, 제한 초과 시 에스컬레이션

### 산출물 저장 구조

```
artifacts/
└── {에픽키}/
    ├── 00_idea/                ← 기획 Phase (Ideator + Planner + Spec)
    │   ├── proposal.md         ← 기획서 원문 (Jira에 올린 내용)
    │   ├── codebase_analysis.md ← 코드 분석 결과
    │   └── idea_report.md      ← 실행 보고서
    └── {스토리키}/              ← 개발 Phase (스토리별)
        ├── 01_designer_1/
        │   └── wireframe.md
        ├── 02_test/
        │   └── test_cases.md
        ├── 03_dev/
        │   └── (코드는 git branch에 직접 반영)
        ├── 04_designer_2/
        │   └── visual_spec.md
        ├── 05_review/
        │   ├── primary_review.md
        │   └── secondary_review.md
        └── 06_release/
            └── release_notes.md
```

### 조건 분기 처리

오케스트레이터가 Jira 메타 필드를 읽어 분기한다:
- `requires_ui_change == yes` → Designer AI 1차/2차 단계 활성화
- `risk_level` → CP 승인 방식 결정 (08_human_checkpoint 참조)
- `system` (백엔드/앱/웹) → 플랫폼별 Agent 동작 분기

도구 연동 상세는 14_tool_integration을 참조한다.

## 커맨드 매핑

| 커맨드 | Phase | Agent | 설명 |
|--------|-------|-------|------|
| `/idea` | 기획 | Ideator + Planner + Spec | 아이디어 → 기획 → 설계 → Jira 생성 |
| `/in-progress` | 개발 | Test + Dev + Review + Release | 스토리 단위 개발 플로우 실행 |
| `/init-project` | 설정 | - | 레포 초기 설정 + 레지스트리 등록 |
| `/init-epic` | 설정 | - | 멀티 레포 에픽 설정 |
| `/status` | 유틸 | - | 진행 상태 확인 |
| `/hotfix` | 긴급 | Dev + Review | 프로덕션 긴급 수정 |
| `/design-proposal` | 유틸 | Designer | UI/UX 시안 생성 |

### 개별 실행 (기존 Jira 티켓이 있는 경우)

`/idea` 없이 사람이 직접 만든 Jira 티켓에 대해서는 개별 커맨드로 실행할 수 있다:
- `/plan` — 요구사항 확장 + 작업 분해
- `/spec` — 설계 문서 생성
- `/test` — 테스트 설계 + 코드
- `/dev` — 코드 구현
- `/review` — 코드 리뷰
- `/release` — 릴리즈
