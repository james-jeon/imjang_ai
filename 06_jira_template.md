# AI 호환 Jira 템플릿

## 작성 원칙

- 필수 항목만 채우면 AI가 나머지를 확장한다
- 의도가 명확할수록 AI 확장 정확도가 높아진다
- 부수적 맥락은 선택 항목에 자유롭게 추가한다

---

## 필수 항목 (사람이 반드시 작성)

### 제목
작업을 한 줄로 요약

### 문제/배경
왜 이 작업이 필요한가 (1~2문장)

### 범위
대상 시스템 또는 플랫폼 (예: Android/iOS, 백엔드, 웹)

### 기대 결과
이 작업이 완료되면 어떤 상태인가 (1~2문장)

---

## 선택 항목 (추가 맥락이 있을 때 작성)

### 원인 상세
문제의 구체적 원인이나 재현 조건

### 구현 방향
선호하는 구현 방식이나 UX 패턴 (예: 토스트, 팝업, 별도 화면)

### 제약 조건
기술적 제약, 일정, 호환성 요구사항

### 관련 자료
참고 문서, 디자인 시안, CS 데이터, 관련 Jira 링크

### 우선순위 사유
왜 지금 해야 하는가 (비즈니스 맥락)

---

## 메타 필드

| 필드 | 필수 여부 | 허용 값 | 기본값 | 설명 |
|------|:---------:|---------|:------:|------|
| type | 필수 | feature / bug / improvement / refactor | - | 작업 유형 |
| system | 필수 | app / web / backend / infra | - | 대상 시스템 |
| risk_level | 필수 | low / medium / high / critical | low | 위험도. Human Checkpoint 승인 방식을 결정한다 (08_human_checkpoint 참조) |
| requires_db_change | 선택 | yes / no | no | DB 스키마 변경 포함 여부. yes인 경우 가드레일 적용 |
| requires_api_change | 선택 | yes / no | no | API 인터페이스 변경 포함 여부 |
| requires_ui_change | 선택 | yes / no | no | UI 변경 포함 여부 |

---

## risk_level 산정 기준

AI가 티켓 생성/확장 시 자동으로 risk_level을 판정한다. 사람은 CP1에서 이를 확인하고 재지정할 수 있다.

### 판정 기준표

| 조건 | risk_level | 이유 |
|------|:----------:|------|
| 결제/과금 로직 변경 | critical | 금전적 영향 |
| 인증/인가 로직 변경 | critical | 보안 영향 |
| 개인정보 처리 변경 | critical | 법적 영향 |
| DB 스키마 변경 (마이그레이션) | high | 롤백 어려움 |
| 외부 API 연동 변경 | high | 외부 의존성 |
| 인프라/배포 설정 변경 | high | 서비스 안정성 |
| 새 API 엔드포인트 추가 | medium | 인터페이스 변경 |
| 기존 비즈니스 로직 수정 | medium | 사이드 이펙트 가능 |
| 멀티 플랫폼 동시 변경 | medium | 일관성 검증 필요 |
| UI 변경 (기존 패턴 내) | low | 영향 범위 제한적 |
| 텍스트/리소스만 변경 | low | 로직 변경 없음 |
| 버그 수정 (단일 모듈) | low | 영향 범위 명확 |

### 복합 조건

여러 조건에 해당하면 **가장 높은 level을 적용**한다.

예: UI 변경(low) + DB 스키마 변경(high) → **high**

### 기본값과 재지정

- 기본값: `low`
- AI가 티켓 내용을 분석하여 자동 판정한다
- 사람은 CP1(요구사항 확인) 또는 CP2(계획 승인) 시점에 재지정할 수 있다
- 재지정 시 Jira 티켓에 사유를 코멘트로 남긴다

---

## requires_ui_change 산정 기준

AI가 티켓 내용에서 다음 조건을 확인하여 자동 판정한다.

| 조건 | 판정 |
|------|:----:|
| 새 화면/페이지 추가 | yes |
| 기존 화면 레이아웃 변경 | yes |
| 새 UI 컴포넌트 추가 | yes |
| 사용자 인터랙션 변경 (버튼, 플로우 등) | yes |
| 에러/안내 메시지 UI 추가 | yes |
| API 로직만 변경 (화면 변화 없음) | no |
| 텍스트 내용만 변경 (레이아웃 동일) | no |
| 백엔드/인프라만 변경 | no |
| 성능 개선 (동작 동일) | no |

기본값: `no`

## AI 역질문 프로세스

AI가 Jira 초안을 분석한 후 필수 항목이 누락되었거나 내용이 모호한 경우, 작업을 시작하지 않고 사람에게 역질문한다.

### 역질문 조건

- 필수 항목(제목, 문제/배경, 범위, 기대 결과) 중 하나라도 비어 있는 경우
- 필수 메타 필드(type, system, risk_level) 중 하나라도 비어 있는 경우
- 필수 항목이 채워져 있으나 해석이 불가능할 정도로 모호한 경우

### 역질문 채널

- Jira 티켓 코멘트에 "추가 정보 필요" 섹션으로 작성한다

### 역질문 형식

```
[추가 정보 필요]

1. "범위"가 비어 있습니다. 대상 시스템을 알려주세요. (app / web / backend / infra)
2. "기대 결과"가 모호합니다. 구체적으로 어떤 상태가 되어야 하는지 알려주세요.
```

### 응답 대기

- 역질문 후 사람이 응답할 때까지 해당 작업을 진행하지 않는다
- 24시간 이내 응답이 없으면 Jira 티켓에 리마인더 코멘트를 추가한다

## AI 확장 영역 (AI가 자동 생성)

AI는 위 입력을 기반으로 다음을 확장한다.

- 상세 요구사항 (Requirements)
- 비기능 요구사항 (Non Functional Requirements)
- 인수 조건 (Acceptance Criteria)
- 영향 범위 (Impact Scope)
- 테스트 전략 (Test Strategy)
- 작업 분해 (Sub Tasks) — **스토리 description 내부에 체크리스트로 포함**. 작업 분해 항목을 별도 Jira 이슈로 생성하지 않는다
- 배포 참고사항 (Deployment Notes)

### 스토리 세분화 원칙

- **에픽 하위 스토리는 플랫폼별(Android/iOS/Backend/Web) 1개씩만 생성**한다
- 작업 분해(Sub Tasks)는 스토리 description의 "작업 내용" 섹션에 bullet list로 포함한다
- 작업 분해 항목 하나하나를 별도 Jira 스토리로 만들지 않는다
- 예: "위젯 추가" 에픽 → "[Android] 위젯 구현" 스토리 1개 + "[iOS] 위젯 구현" 스토리 1개

## 예시 1: feature (앱 UX)

### 사람 작성 (초안)

```
제목: 출입 인증 실패 시 안내 UX 추가

문제/배경: 인증 실패 시 원인을 알 수 없어 CS 문의가 많음

범위: Android / iOS 앱

기대 결과: 인증 실패 시 원인별 안내 메시지가 표시된다

구현 방향: 기존 토스트 패턴 활용 (선택)
원인 상세: BLE OFF, 위치권한 OFF 등 설정 문제가 주요 원인 (선택)

메타 필드:
  type: feature
  system: app
  risk_level: low
  requires_ui_change: yes
```

### AI 확장 (자동 생성)

```
요구사항:
- BLE 비활성화 시 "블루투스를 켜주세요" 안내
- 위치권한 미허용 시 "위치 권한을 허용해주세요" 안내
- 네트워크 오류 시 "네트워크 연결을 확인해주세요" 안내
- 원인 불명 시 "고객센터에 문의해주세요" 안내

인수 조건:
- 각 실패 원인별 안내 메시지가 정확히 표시된다
- 안내 메시지에서 설정 화면으로 이동할 수 있다
- 기존 인증 성공 플로우에 영향 없다

영향 범위:
- AuthenticationManager
- EntryFailureHandler
- strings.xml / Localizable.strings

테스트 전략:
- BLE OFF 상태에서 인증 시도 → 안내 표시 확인
- 위치권한 OFF → 안내 표시 확인
- 네트워크 OFF → 안내 표시 확인
- 정상 상태 → 기존 플로우 정상 동작 확인

작업 분해:
1. 인증 실패 원인 분류 로직 추가
2. 원인별 메시지 리소스 추가
3. 토스트 표시 로직 구현
4. 설정 화면 이동 딥링크 처리
5. 단위 테스트 작성
```

## 예시 2: bug (백엔드 API)

### 사람 작성 (초안)

```
제목: 예약 목록 API에서 취소된 예약이 조회되는 버그

문제/배경: 사용자가 취소한 예약이 목록에 계속 노출됨

범위: 백엔드

기대 결과: 취소된 예약은 목록 API 응답에 포함되지 않는다

원인 상세: status 필터 없이 전체 조회하고 있는 것으로 추정 (선택)

메타 필드:
  type: bug
  system: backend
  risk_level: medium
  requires_api_change: no
```

### AI 확장 (자동 생성)

```
요구사항:
- GET /api/reservations 응답에서 status=CANCELLED인 항목 제외
- 기존 관리자 API는 취소 건 포함하여 조회 유지

인수 조건:
- 예약 취소 후 목록 재조회 시 해당 예약이 보이지 않는다
- 관리자 화면에서는 취소된 예약이 정상 조회된다
- 페이지네이션에 영향 없다

영향 범위:
- ReservationRepository.findByUserId()
- ReservationService.getList()

테스트 전략:
- 취소된 예약 포함 데이터 세팅 → 목록 API 호출 → 취소 건 미포함 확인
- 관리자 API 호출 → 취소 건 포함 확인
- 빈 목록 케이스 → 정상 응답 확인

작업 분해:
1. ReservationRepository 쿼리에 status 필터 추가
2. 기존 테스트 수정
3. 신규 테스트 케이스 추가
```

## 예시 3: refactor (백엔드 성능)

### 사람 작성 (초안)

```
제목: 대시보드 통계 API 응답 속도 개선

문제/배경: 대시보드 로딩에 5초 이상 걸려 사용성 저하

범위: 백엔드

기대 결과: 대시보드 통계 API 응답이 1초 이내로 개선된다

제약 조건: 기존 응답 스펙 변경 불가 (선택)
원인 상세: 매 요청마다 집계 쿼리를 실행하고 있음 (선택)

메타 필드:
  type: refactor
  system: backend
  risk_level: high
  requires_db_change: yes
```

### AI 확장 (자동 생성)

```
요구사항:
- 통계 데이터를 사전 집계하여 캐시 또는 집계 테이블에 저장
- API는 집계된 결과를 조회만 수행
- 집계 주기: 5분 간격 배치 또는 이벤트 기반 갱신

비기능 요구사항:
- 응답 시간 1초 이내 (p95 기준)
- 집계 데이터와 실시간 데이터 간 최대 5분 지연 허용

인수 조건:
- 대시보드 API 응답 시간이 1초 이내이다
- 기존 응답 형식과 동일하다
- 집계 배치 실패 시 마지막 성공 데이터를 반환한다

영향 범위:
- DashboardStatisticsService
- StatisticsRepository
- 신규: statistics_summary 테이블
- 신규: StatisticsAggregationJob

테스트 전략:
- 대량 데이터(10만건) 기준 응답 시간 측정
- 집계 배치 정상 동작 확인
- 집계 실패 시 fallback 동작 확인
- 기존 API 응답 형식 호환성 확인

작업 분해:
1. statistics_summary 테이블 설계 및 마이그레이션
2. 집계 배치 Job 구현
3. DashboardStatisticsService를 집계 테이블 조회로 변경
4. fallback 로직 구현
5. 성능 테스트 작성
6. 기존 단위 테스트 수정

배포 참고사항:
- DB 마이그레이션 선행 배포 필요
- 배치 Job 등록 필요
- risk_level=high → 사람 리뷰 필수
```
