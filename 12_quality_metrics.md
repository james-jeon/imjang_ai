# AI Quality Metrics

## 목적

AI가 만든 결과물의 품질을 측정하고 추적하는 기준을 정의한다.
"적을수록 좋음" 같은 추상적 기준이 아니라, **baseline → 임계치 → 경고 조건**으로 운영한다.

## Baseline 수집 방법

### 초기 2주 (첫 도입 시)

1. **첫 3개 티켓**을 `/in-progress`로 실행한다
2. 각 티켓에서 다음을 기록한다:
   - `process_log.md`의 토큰 사용량
   - 리뷰 라운드 수
   - AC 충족률 (1차 리뷰 기준)
   - 테스트 보완 발생 여부
   - PR에서 사람 수정 건수
3. 3개 티켓의 평균값을 Baseline으로 설정한다

### 데이터 소스

| 지표 | 데이터 소스 |
|------|-----------|
| 테스트 통과율 | `change_summary.md` → [테스트 실행 결과] 섹션 |
| AC 구현 일치율 | `primary_review.md` → [요구사항 충족] 테이블 |
| 리뷰 반려율 | `review_summary.md` → [최종 판정] |
| 토큰 사용량 | `process_log.md` → [토큰 사용량 분석] |
| PR 수정 건수 | GitHub PR → 머지 후 커밋 이력 |

## 경고 및 대응

### 경고 발생 시 대응 절차

| 심각도 | 조건 | 대응 |
|--------|------|------|
| **주의** | 지표 1개가 경고 임계치 도달 | 다음 티켓에서 해당 영역 모니터링 강화 |
| **경고** | 지표 2개 이상 경고 또는 1개가 2회 연속 | 프로세스 점검 — 원인 분석 후 Agent 프롬프트/게이트 규칙 수정 |
| **중단** | Critical 지표(빌드 성공률, AC 일치율)가 임계치 미달 | `/in-progress` 실행 중단, 프로세스 재설계 |

### 구체적 대응 예시

| 상황 | 원인 | 대응 |
|------|------|------|
| 테스트 보완 라운드 70% 이상 | Test Agent가 placeholder/부실 테스트 생성 | tester.md 품질 기준 강화, Test Validator 게이트 추가 |
| 리뷰 반려율 50% 이상 | Dev Agent가 AC를 누락 | Dev Agent 프롬프트에 AC 체크리스트 강제 포함 |
| 토큰 800K/story 이상 | Jira 조회 분산, 코드 탐색 과다 | Jira 통합 조회, 레포 사전 스캔 적용 |
| PR 수정 5건 이상 | AI 코드 품질 부족 | 리뷰 게이트 강화, 3차 리뷰 추가 검토 |
