## 1. 요구사항 상세

### 1.1 인증 (Auth)

**FR-AUTH-01: 이메일/비밀번호 회원가입**
- 사용자는 이메일과 비밀번호로 회원가입할 수 있다.
- AC:
  - 유효한 이메일 형식 + 비밀번호 8자 이상 입력 시 가입 성공, 홈 화면으로 이동한다
  - 이미 등록된 이메일 입력 시 "이미 가입된 이메일입니다" 에러 메시지가 표시된다
  - 이메일 형식 불일치 시 "올바른 이메일 형식이 아닙니다" 에러가 표시된다
  - 비밀번호 8자 미만 시 "비밀번호는 8자 이상이어야 합니다" 에러가 표시된다
  - 비밀번호 확인 입력이 비밀번호와 불일치 시 "비밀번호가 일치하지 않습니다" 에러가 표시된다

**FR-AUTH-02: 이메일/비밀번호 로그인**
- 사용자는 이메일과 비밀번호로 로그인할 수 있다.
- AC:
  - 올바른 이메일/비밀번호 입력 시 로그인 성공, 홈 화면으로 이동한다
  - 틀린 이메일/비밀번호 입력 시 "이메일 또는 비밀번호가 일치하지 않습니다" 에러가 표시된다
  - 로그인 상태는 앱 재시작 후에도 유지된다 (Firebase Auth persistence)

**FR-AUTH-03: Google 소셜 로그인**
- 사용자는 Google 계정으로 로그인할 수 있다.
- AC:
  - Google 로그인 버튼 탭 시 Google 인증 화면이 표시된다
  - Google 인증 완료 시 앱 홈 화면으로 이동한다
  - 최초 Google 로그인 시 Firestore `users` 컬렉션에 사용자 문서가 생성된다

**FR-AUTH-04: Apple 소셜 로그인 (iOS)**
- iOS 사용자는 Apple 계정으로 로그인할 수 있다.
- AC:
  - iOS에서 Apple 로그인 버튼이 표시된다
  - Apple 인증 완료 시 앱 홈 화면으로 이동한다
  - Android에서는 Apple 로그인 버튼이 표시되지 않는다

**FR-AUTH-05: 로그아웃**
- 사용자는 설정 화면에서 로그아웃할 수 있다.
- AC:
  - 로그아웃 버튼 탭 시 확인 다이얼로그가 표시된다
  - 확인 시 로그인 화면으로 이동하고 인증 상태가 해제된다

**FR-AUTH-06: 사용자 프로필 관리**
- 사용자는 닉네임과 프로필 사진을 설정할 수 있다.
- AC:
  - 닉네임 입력란에 2~20자 닉네임을 입력하고 저장할 수 있다
  - 프로필 사진을 갤러리에서 선택하거나 카메라로 촬영하여 설정할 수 있다
  - 프로필 사진은 256x256으로 리사이즈 후 Storage에 업로드된다

### 1.2 단지 관리 (Complex)

**FR-COMP-01: 단지 등록 (공공API 검색)**
- 사용자는 아파트 단지명을 검색하여 공공데이터에서 조회된 단지를 등록할 수 있다.
- AC:
  - 단지명 2글자 이상 입력 시 500ms 디바운스 후 자동완성 목록이 표시된다
  - 자동완성 목록에는 단지명, 주소, 세대수가 표시된다
  - 자동완성 항목 선택 시 공공API에서 단지 상세정보를 자동으로 가져온다
  - 등록 완료 시 단지 상세 화면으로 이동한다
  - 이미 등록된 단지를 다시 검색한 경우 "이미 등록된 단지입니다" 안내와 함께 기존 단지로 이동 옵션이 표시된다

**FR-COMP-02: 단지 기본 정보 표시**
- 등록된 단지의 기본 정보를 표시한다.
- AC:
  - 다음 정보가 표시된다: 단지명, 주소(도로명+지번), 세대수, 동수, 최저/최고층, 난방방식, 사용승인일, 건설사, 용적률/건폐율
  - 공공API에서 정보를 가져올 수 없는 경우 "정보 없음"으로 표시된다

**FR-COMP-03: 단지 상태 관리**
- 사용자는 단지의 상태를 변경할 수 있다.
- AC:
  - 상태 옵션: 관심, 방문 예정, 방문 완료, 재방문, 제외
  - 상태 변경 시 변경 일시가 기록된다
  - 단지 목록에서 상태별 아이콘/색상이 구분되어 표시된다

**FR-COMP-04: 단지 목록 조회 + 필터**
- 사용자가 등록한 모든 단지를 목록으로 조회하고, 조건별 필터링할 수 있다.
- AC:
  - 단지명, 주소, 상태, 최근 임장일, 대표 면적, 최근 실거래가가 목록에 표시된다
  - **상태 필터**: 전체/관심/방문예정/방문완료/재방문/제외 (필터 칩)
  - **매매가 필터**: 가격 범위 슬라이더 (1억~30억, 1억 단위). 최근 6개월 실거래 기준
  - **면적(평수) 필터**: 10평대 이하 / 20평대 / 30평대 / 40평대 / 50평대 이상 (다중 선택 칩). 전용면적 기준
  - **세대수 필터**: 전체 / 300세대 이상 / 500세대 이상 / 1,000세대 이상 (단일 선택)
  - **준공연도 필터**: 전체 / 5년 이내 / 10년 이내 / 15년 이내 / 20년 이내 / 20년 초과 (단일 선택). 사용승인일 기준
  - **임장 평점 필터**: 전체 / ★3 이상 / ★4 이상 (단일 선택). 종합 평점 기준
  - 필터는 목록 상단 필터 칩 바 + "상세 필터" 바텀시트로 제공된다
  - 필터 칩 바에는 상태/면적을 기본 표시하고, 나머지는 바텀시트에서 설정한다
  - 활성화된 필터 개수가 배지로 표시된다
  - 필터 조합은 AND 조건으로 동작한다
  - 필터 결과가 0건이면 "조건에 맞는 단지가 없습니다" + 필터 초기화 버튼이 표시된다
  - 정렬 옵션: 최근 등록순, 이름순, 최근 임장순, 매매가 낮은순, 매매가 높은순
  - 단지 20개 초과 시 무한 스크롤로 추가 로딩된다
  - **구현 방식**: 클라이언트 사이드 필터링 (MVP 규모에서 사용자당 단지 수 제한적이므로 Firestore 복합 쿼리 제약 회피)

**FR-COMP-05: 단지 삭제**
- 사용자는 등록한 단지를 삭제할 수 있다.
- AC:
  - 삭제 버튼 탭 시 "단지를 삭제하면 관련 임장 기록도 모두 삭제됩니다" 경고 다이얼로그가 표시된다
  - 확인 시 단지와 연결된 모든 임장 기록, 사진이 삭제된다
  - 공유된 단지는 Owner만 삭제할 수 있다

**FR-COMP-06: 실거래가 자동 조회**
- 단지 등록/조회 시 최근 실거래가를 자동으로 조회하여 표시한다.
- AC:
  - 단지 상세 화면에 최근 6개월 실거래가 목록이 표시된다
  - 각 거래 항목에 계약일, 면적(평형), 가격, 층이 표시된다
  - 실거래가 데이터가 없는 경우 "최근 6개월 거래 내역이 없습니다" 메시지가 표시된다
  - 실거래가 새로고침 버튼이 제공된다
  - 공공API 호출 실패 시 캐시된 이전 데이터가 표시되고, "최신 데이터를 불러올 수 없습니다" 토스트가 표시된다

### 1.3 임장 기록 (Inspection)

**FR-INSP-01: 임장 기록 작성**
- 사용자는 단지에 대한 임장 기록을 작성할 수 있다.
- AC:
  - 다음 필드를 입력할 수 있다:
    - 방문 날짜 (DatePicker, 기본값: 오늘)
    - 방문 시간대 (아침/오전/오후/저녁/야간, 다중 선택 가능)
    - 체크항목 평점 (1~5): 소음, 경사, 상권, 주차, 일조권
    - 장점 (텍스트, 최대 1000자)
    - 단점 (텍스트, 최대 1000자)
    - 총평 (텍스트, 최대 2000자)
    - 종합 평점 (1~5, 0.5 단위)
  - 체크항목 평점은 별점 또는 슬라이더 UI로 입력한다
  - 저장 시 작성일시, 작성자 정보가 자동 기록된다

**FR-INSP-02: 임장 기록에 사진 첨부**
- 사용자는 임장 기록에 사진을 첨부할 수 있다.
- AC:
  - 카메라 촬영 또는 갤러리에서 사진을 선택할 수 있다
  - 1회 기록당 최대 20장까지 첨부 가능하다
  - 각 사진에 메모(캡션)를 추가할 수 있다 (최대 200자)
  - 사진은 업로드 전 장축 기준 1920px, 품질 80%로 자동 압축된다
  - 업로드 진행률이 표시된다

**FR-INSP-03: 임장 기록 조회**
- 사용자는 단지의 임장 기록을 조회할 수 있다.
- AC:
  - 단지 상세 화면에서 해당 단지의 모든 임장 기록 목록이 표시된다
  - 각 기록에 방문일, 시간대, 종합 평점, 사진 썸네일(첫 번째)이 표시된다
  - 기록 탭 시 상세 화면으로 이동하여 모든 입력 항목 + 사진이 표시된다

**FR-INSP-04: 임장 기록 수정**
- 사용자는 작성한 임장 기록을 수정할 수 있다.
- AC:
  - 모든 입력 필드를 수정할 수 있다
  - 사진 추가/삭제가 가능하다
  - 수정일시가 자동 기록된다
  - 공유된 기록의 경우 Editor 이상 권한에서만 수정 가능하다

**FR-INSP-05: 임장 기록 삭제**
- 사용자는 작성한 임장 기록을 삭제할 수 있다.
- AC:
  - 삭제 시 확인 다이얼로그가 표시된다
  - 확인 시 기록과 연결된 사진이 Storage에서도 삭제된다

**FR-INSP-06: 오프라인 임장 기록 작성**
- 네트워크 연결 없이도 임장 기록을 작성할 수 있다.
- AC:
  - 오프라인 상태에서 기록 작성 및 저장이 가능하다
  - 오프라인 상태에서 사진 촬영 및 로컬 임시 저장이 가능하다 (앱 documents 디렉토리에 저장)
  - 오프라인 상태 표시 아이콘이 앱 상단에 표시된다
  - 네트워크 복구 시 텍스트 데이터는 Firestore 오프라인 캐시가 자동 동기화한다
  - 네트워크 복구 시 사진은 앱 자체 업로드 큐를 통해 순차적으로 Firebase Storage에 업로드된다 (Firebase Storage는 오프라인 자동 큐잉을 지원하지 않으므로 수동 구현 필요)
  - 사진 업로드 실패 시 최대 3회 재시도 후 실패 상태로 전환되며, 사용자가 수동 재시도 버튼으로 다시 시도할 수 있다
  - 동기화 완료/실패 시 사용자에게 알림(토스트)이 표시된다

### 1.4 지도 (Map)

**FR-MAP-01: 지도 화면 표시**
- Kakao Map을 이용하여 지도 화면을 표시한다.
- AC:
  - 앱 하단 탭에서 지도 탭 선택 시 카카오 지도가 전체 화면으로 표시된다
  - 초기 위치는 사용자의 현재 위치 또는 마지막으로 본 위치이다
  - 현재 위치 버튼 탭 시 현재 GPS 위치로 이동한다
  - 줌 레벨 조절이 가능하다 (핀치/버튼)
  - 위치 권한 거부 시 현재 위치 버튼은 비활성화되고, 지도는 서울 시청(기본 위치)을 중심으로 표시된다. "위치 권한이 필요합니다" 안내 메시지가 표시된다

**FR-MAP-02: 단지 마커 표시**
- 등록된 단지를 지도 위에 마커(핀)로 표시한다.
- AC:
  - 등록된 모든 단지가 해당 위치에 마커로 표시된다
  - 마커 색상은 단지 상태에 따라 구분된다:
    - 관심: 노랑
    - 방문 예정: 파랑
    - 방문 완료: 초록
    - 재방문: 주황
    - 제외: 회색
  - 마커 탭 시 말풍선(InfoWindow)에 단지명, 상태, 종합 평점이 표시된다
  - 말풍선 탭 시 단지 상세 화면으로 이동한다

**FR-MAP-03: 지도 필터**
- 지도에서 단지를 조건별로 필터링할 수 있다.
- AC:
  - 지도 상단에 상태 필터 칩(Chip)이 표시된다
  - 필터 칩 선택/해제 시 해당 상태의 마커가 표시/숨김된다
  - "상세 필터" 버튼 탭 시 바텀시트가 표시되며, FR-COMP-04와 동일한 필터 항목(매매가/면적/세대수/준공연도/임장 평점)을 적용할 수 있다
  - 단지 목록과 지도의 필터 상태는 동기화된다 (한쪽에서 변경하면 다른 쪽에도 반영)
  - 필터 상태는 화면 전환 후에도 유지된다

**FR-MAP-04: 지도에서 단지 검색**
- 지도 화면에서 단지를 검색하여 해당 위치로 이동할 수 있다.
- AC:
  - 지도 상단 검색창에 단지명 입력 시 등록된 단지 중 매칭되는 결과가 표시된다
  - 검색 결과 선택 시 해당 단지 위치로 지도가 이동하고 마커가 강조된다

### 1.5 공유/공동편집 (Share)

**FR-SHARE-01: 공유 링크 생성**
- 단지 Owner는 공유 링크를 생성하여 다른 사용자를 초대할 수 있다.
- AC:
  - 단지 상세 화면에서 "공유" 버튼 탭 시 공유 설정 화면이 표시된다
  - 링크 생성 시 권한(Editor/Viewer)을 선택할 수 있다
  - 생성된 링크를 클립보드에 복사하거나 OS 공유 시트로 공유할 수 있다
  - 공유 링크는 생성 후 7일이 지나면 만료된다. 만료된 링크로 접근 시 "초대 링크가 만료되었습니다" 메시지가 표시된다
  - 링크는 App Links / Universal Links 딥링크로 구현한다

**FR-SHARE-02: 공유 초대 수락**
- 공유 링크를 받은 사용자는 로그인 후 해당 단지에 접근할 수 있다.
- AC:
  - 공유 링크 클릭 시 앱이 열리고 (미설치 시 스토어로 이동), 로그인 후 해당 단지가 내 목록에 추가된다
  - 비로그인 사용자는 로그인 화면으로 이동 후 자동으로 공유 단지에 연결된다

**FR-SHARE-03: 권한 관리 (Owner/Editor/Viewer)**
- Owner는 공유 참여자의 권한을 관리할 수 있다.
- AC:
  - 권한 종류:
    - Owner: 모든 권한 + 삭제 + 권한 관리
    - Editor: 임장 기록 작성/수정, 단지 상태 변경
    - Viewer: 읽기 전용
  - Owner는 참여자의 권한을 변경하거나 제거할 수 있다
  - Editor는 Viewer를 Editor로 승격할 수 없다 (Owner만 가능)

**FR-SHARE-04: 실시간 동기화**
- 공유된 단지의 데이터가 참여자 간 실시간으로 동기화된다.
- AC:
  - 한 사용자가 임장 기록을 추가/수정하면 다른 참여자의 화면에 5초 이내에 반영된다 (Wi-Fi 환경 기준. LTE/3G에서는 지연이 더 클 수 있으며, 이는 공식 Firebase SLA가 아닌 목표치이다)
  - 동시 편집 시 "마지막 쓰기 우선(Last Write Wins)" 정책을 따른다
  - 필드 레벨 병합: 서로 다른 필드를 수정한 경우 양쪽 변경 모두 반영된다

**FR-SHARE-05: 활동 로그**
- 공유 단지에서의 활동 내역을 확인할 수 있다.
- AC:
  - 다음 활동이 로그에 기록된다: 기록 작성, 기록 수정, 상태 변경, 참여자 추가/제거
  - 각 로그에 수행자, 일시, 내용이 표시된다
  - 최근 50건의 활동 로그가 표시되며 이전 기록은 "더 보기"로 로딩된다

### 1.6 공공데이터 연동 (PublicAPI)

**FR-API-01: 공동주택 단지 목록 조회**
- 법정동코드를 기반으로 공동주택 단지 목록을 조회한다.
- AC:
  - 지역(시/도 > 시/군/구 > 읍/면/동) 선택 시 해당 지역의 아파트 단지 목록이 반환된다
  - 단지명으로 텍스트 검색이 가능하다
  - API 응답 시간이 3초를 초과하면 "조회 중..." 로딩 표시 후, 10초 초과 시 타임아웃 처리하고 "잠시 후 다시 시도해주세요" 메시지를 표시한다

**FR-API-02: 공동주택 기본 정보 조회**
- 선택된 단지의 상세 정보를 조회한다.
- AC:
  - 단지 등록 시 자동으로 호출되어 세대수, 동수, 최저/최고층, 난방방식, 사용승인일, 건설사 등을 가져온다
  - 조회된 정보는 Firestore에 캐싱되어 이후 재조회 시 API를 호출하지 않는다
  - 캐시 갱신 주기: 30일 (수동 새로고침 가능)

**FR-API-03: 아파트 매매 실거래가 조회**
- 단지의 매매 실거래가를 조회한다.
- AC:
  - 법정동코드 + 계약년월(최근 6개월)로 조회한다
  - 조회 결과를 Firestore에 캐싱하며, 캐시 유효기간은 7일이다
  - 동일 법정동 내 다른 단지의 데이터도 함께 캐싱한다 (API가 법정동 단위로 반환하므로)

**FR-API-04: 건축물대장 정보 조회**
- 단지의 건축물대장 정보를 조회한다.
- AC:
  - 시군구코드 + 번/지로 건축물대장 정보를 조회한다
  - 대지면적, 건축면적, 연면적, 사용승인일 등을 가져온다
  - 조회 결과는 Firestore에 영구 캐싱한다 (건축물대장은 거의 변경되지 않으므로)

**FR-API-05: 법정동코드 로컬 캐시**
- 법정동코드를 로컬 DB에 캐싱하여 오프라인에서도 지역 검색이 가능하다.
- AC:
  - 앱 최초 실행 시 법정동코드 데이터(약 5,000건)를 로컬 DB에 저장한다
  - 시/도 > 시/군/구 > 읍/면/동 계층 구조로 검색 가능하다
  - 법정동코드 데이터는 앱 번들에 포함하거나 최초 실행 시 다운로드한다
  - 데이터 갱신 주기: 앱 업데이트 시 (법정동코드는 연 1~2회 변경)

## 5. risk_level 판정

| 영역 | risk_level | 판정 근거 |
|------|:----------:|-----------|
| **Auth** | low | Firebase Auth는 성숙한 솔루션. Google/Apple 로그인은 공식 플러그인으로 표준화. 구현 패턴 풍부 |
| **Core** | low | Flutter 프로젝트 초기화, go_router, Riverpod은 잘 문서화된 표준 구성 |
| **Complex (단지 관리)** | low | 표준 Firestore CRUD. 데이터 모델이 단순(NoSQL 문서) |
| **Inspection (임장 기록)** | low~medium | 기본 CRUD는 low. 사진 업로드/압축/진행률은 medium (패키지 조합 필요) |
| **Map (지도)** | medium | Kakao Map API 무료 할당량(일 30만건) 관리 필요. 마커 100개 이상 성능 미확인. 할당량 초과 시 캐싱으로 완화 |
| **PublicAPI (공공데이터)** | medium | 4개 외부 API 연동. XML 응답 파싱 필요. API 장애/포맷 변경 리스크. 호출 제한 관리 필요 |
| **Share (공유/공동편집)** | high | 실시간 동기화 + 권한 관리 + 충돌 해결 + Security Rules 설계. Firestore Security Rules 설계 오류 시 데이터 유출 위험. 딥링크 연동 복잡도 |
| **Offline (오프라인)** | medium | Firestore 기본 오프라인은 자동이나, 사진 로컬 임시 저장 + 동기화 복구는 추가 로직 필요 |

## Decision Records

### DR-01: 오프라인 사진 저장 전략

| 항목 | 내용 |
|------|------|
| 결정 사항 | 오프라인 사진을 앱 임시 디렉토리에 저장하고 온라인 복귀 시 Storage 업로드 |
| 대안 A | 앱 임시 디렉토리 저장 + 연결 복구 시 순차 업로드 |
| 대안 B | SQLite BLOB 저장 (사진을 DB에 직접 저장) |
| 대안 C | Firestore offline cache에 의존 (사진 URL만 저장, 실제 파일은 별도 처리 안 함) |
| 채택 사유 | 대안 A: 파일 시스템이 대용량 바이너리에 최적. path_provider로 임시 경로 접근 간단 |
| 기각 사유 | 대안 B: DB에 바이너리 저장은 DB 크기 급증 + 성능 저하. 대안 C: Firestore는 파일 저장 불가, Storage 업로드는 별도 로직 필수 |
| 제약/한계 | 앱 삭제 시 임시 저장 사진 유실. 앱 재설치 전 동기화 미완료 사진 경고 필요 |

### DR-02: 공공데이터 API 캐싱 전략

| 항목 | 내용 |
|------|------|
| 결정 사항 | Firestore에 API 응답을 캐싱 (컬렉션: `apiCache`) |
| 대안 A | Firestore 캐싱 (서버 사이드, 모든 디바이스 공유) |
| 대안 B | 로컬 DB (Hive/SQLite) 캐싱 (디바이스별 독립) |
| 대안 C | Firebase Cloud Functions에서 주기적으로 API 호출 + Firestore에 저장 |
| 채택 사유 | 대안 A: 한 사용자가 조회한 데이터를 다른 사용자도 활용. 공공API 호출 횟수 절감. 오프라인에서도 Firestore 캐시로 접근 가능 |
| 기각 사유 | 대안 B: 디바이스마다 중복 호출. 공유 사용자 간 캐시 불가. 대안 C: Cloud Functions 비용 + 설정 복잡도 증가. MVP 과잉 |
| 제약/한계 | Firestore 쓰기 비용 발생 (무료 할당량 내 가능). 캐시 무효화 로직 필요 |

### DR-03: 동시 편집 충돌 해결 전략

| 항목 | 내용 |
|------|------|
| 결정 사항 | MVP에서는 Last Write Wins (LWW) + 필드 레벨 병합 |
| 대안 A | LWW + 필드 레벨 병합 (Firestore merge) |
| 대안 B | OT (Operational Transformation) 기반 실시간 충돌 해결 |
| 대안 C | 비관적 잠금 (한 사용자만 편집 가능) |
| 채택 사유 | 대안 A: Firestore `set(merge: true)` / `update()`로 자연스럽게 구현. 서로 다른 필드 편집 시 양쪽 반영. MVP 범위에서 충분 |
| 기각 사유 | 대안 B: 구현 복잡도 극히 높음. Google Docs 수준이며 MVP 과잉. 대안 C: UX 저하 (편집 대기). 2인 이상 동시 사용 시 병목 |
| 제약/한계 | 동일 필드 동시 수정 시 나중 쓰기만 남음. V2에서 충돌 알림/머지 UI 검토 |

### DR-04: 법정동코드 저장 방식

| 항목 | 내용 |
|------|------|
| 결정 사항 | 앱 assets에 JSON/CSV 파일 번들 + 로컬 SQLite에 로딩 |
| 대안 A | 앱 assets 번들 + SQLite |
| 대안 B | 최초 실행 시 서버에서 다운로드 + 로컬 저장 |
| 대안 C | 매번 API 호출 (행정표준코드관리시스템) |
| 채택 사유 | 대안 A: 법정동코드 약 5,000건은 소용량(~500KB). 앱에 포함하면 설치 즉시 사용 가능. 오프라인 완벽 지원 |
| 기각 사유 | 대안 B: 최초 실행 시 네트워크 필수. 다운로드 실패 시 앱 사용 불가. 대안 C: API 호출 제한 + 매번 네트워크 필요. 오프라인 불가 |
| 제약/한계 | 법정동코드 변경(연 1~2회) 시 앱 업데이트 필요. 앱 번들 크기 ~500KB 증가 |
