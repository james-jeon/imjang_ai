# AI 개발 프로세스

## AI 완전 자동화 원칙

**사람은 "확인"과 "판단"만 한다. 실행은 전부 AI가 한다.**

AI가 코드 작성, 파일 생성뿐 아니라 IDE 프로젝트 설정, 빌드 구성, 타겟 추가 등도 직접 수행한다. "이건 수동으로 해야 합니다"라고 사람에게 넘기기 전에, 도구나 라이브러리를 활용한 자동화를 먼저 시도해야 한다.

## 최신 OS 지원 원칙

**각 플랫폼의 최신 릴리즈 OS에서 반드시 동작해야 한다.**

- deprecated된 API, 프레임워크, Extension 타입을 사용하지 않는다
- 기능 설계 시 최신 OS 지원 여부를 **가장 먼저** 확인한다
- 최신 OS에서 제거된 기능은 대체 방식으로 구현한다

| 플랫폼 | 확인 사항 | 예시 |
|--------|----------|------|
| iOS | deprecated 프레임워크/API 사용 금지 | Today Extension(NCWidgetProviding) → WidgetKit + App Intent |
| iOS | 최소 지원 버전에서 사용 가능한 API만 사용 | Interactive Widget은 iOS 17+ |
| Android | targetSdkVersion 최신 유지 | Android 14+ 정책 변경 반영 |
| Android | deprecated API 사용 금지 | IntentService → ForegroundService |
| 공통 | 설계 문서에 지원 OS 범위 명시 | "iOS 17+, Android 12+" |

### AI가 전부 하는 것

| 작업 | 방법 |
|------|------|
| iOS 프로젝트 타겟/Extension 추가 | 빌드 시스템 도구로 pbxproj 수정 |
| iOS Entitlements/Capability 설정 | pbxproj + entitlements 파일 직접 수정 |
| iOS 프레임워크 링크 | 빌드 시스템 도구로 빌드 설정 변경 |
| Android Manifest/Gradle 수정 | XML/Groovy 직접 편집 |
| 빌드 설정 변경 | 플랫폼별 빌드 설정 파일 편집 |
| CI/CD 파이프라인 설정 | YAML/스크립트 직접 생성 |
| 코드 서명 설정 | 빌드 시스템 도구로 빌드 설정 변경 |

> 플랫폼별 구체적 도구와 실행 방법은 **14_tool_integration.md** 참조

### 사람에게 넘기는 기준

**물리적으로 AI가 접근 불가능한 경우만:**

- Apple Developer / Google Play 계정 로그인 (웹 브라우저/계정 인증 필요)
- 실제 기기에서 직접 테스트 (시뮬레이터/에뮬레이터는 AI가 가능)
- 스토어 심사 제출 (계정 인증 필요)
- 프로토타입 확인 (사람의 방향 판단이 필요한 경우)

## 자동 디버깅 원칙

**에러 발생 시 추측이 아닌 로그 기반으로 원인을 분석한다.**

빌드 성공 후 실제 디바이스/시뮬레이터에서 동작하지 않는 경우, AI는 체계적인 디버깅 절차를 따른다. 재부팅, 캐시 삭제 등 근거 없는 시도를 하지 않는다.

### 디버깅 3원칙

| 원칙 | 설명 |
|------|------|
| **로그 우선 (Log-First)** | 에러 발생 시 logcat/콘솔 로그를 **가장 먼저** 수집·분석한다. 추측성 시도(재부팅, 캐시 삭제, 재설치 등)를 하기 전에 반드시 로그를 확인한다 |
| **디버그 로그 선행 삽입** | 프로토타입/구현 시 모든 lifecycle 메서드에 디버그 로그를 포함한다. 문제 발생 후 로그를 추가하는 것이 아니라, 처음부터 삽입한다 |
| **플랫폼 호환성 사전 확인** | 각 플랫폼/벤더(삼성 OneUI, 샤오미 MIUI 등)의 알려진 호환성 이슈를 사전에 확인하고 대응한다 |

### 자동 검증 절차

빌드 성공 후 디바이스 설치 시, AI는 다음 절차를 자동으로 수행한다:

| 단계 | 작업 | 도구 |
|------|------|------|
| 1. 설치 후 로그 모니터링 | 앱/위젯 실행 직후 로그를 실시간 수집 | `adb logcat` / Xcode Console |
| 2. 에러 키워드 필터링 | `Error`, `Exception`, `crash`, `inflate`, `denied` 등 자동 필터링 | grep 패턴 |
| 3. 에러 발견 시 근본 원인 분석 | 로그 메시지 기반으로 코드/리소스/설정 문제 판별 | 로그 → 코드 역추적 |
| 4. 수정 후 재검증 | 수정 적용 후 동일 검증 절차 반복 | 자동화 루프 |

### 플랫폼별 디버깅 체크리스트

| 플랫폼 | 확인 사항 |
|--------|----------|
| Android | `adb logcat` 에러 필터링, `dumpsys appwidget`/`dumpsys package` 상태 확인, RemoteViews inflate 에러 확인 |
| Android | 삼성 OneUI 위젯 호환성: `LinearLayout` + `layout_weight` 대신 `RelativeLayout` 사용, `Button` 대신 `TextView` + clickListener |
| iOS | Xcode 콘솔 로그, Extension 메모리 제한(~30MB) 확인, App Groups 설정 확인 |
| iOS | WidgetKit: extension 프로세스에서 무거운 SDK 로드 금지, App Intent로 메인 앱 프로세스 활용 |

### AI가 전부 하는 것

| 작업 | 방법 |
|------|------|
| 디버그 로그 삽입 | 프로토타입 코드 생성 시 모든 콜백/lifecycle에 `Log.d`/`print` 포함 |
| 빌드 후 로그 수집 | 설치 직후 `adb logcat -c && adb logcat` 또는 Xcode 콘솔 자동 모니터링 |
| 에러 분석 | 로그 메시지에서 에러 패턴 추출 → 코드/리소스 역추적 → 수정안 도출 |
| 플랫폼 호환성 검증 | 벤더별 알려진 이슈 확인 후 레이아웃/API 사전 대응 |

### 사람에게 넘기는 기준

| 상황 | 이유 |
|------|------|
| 실제 기기에서만 재현되는 하드웨어 의존 문제 | AI가 물리적 기기를 조작할 수 없음 |
| 로그에 에러가 없지만 동작이 비정상인 경우 | 사람의 화면 확인 및 설명이 필요 |
| 계정/인증 관련 권한 문제 | AI가 계정 접근 불가 |

## 진입점

프로세스는 두 가지 진입점을 가진다. 상황에 따라 적절한 흐름으로 진입한다.

| 상황 | 진입점 | 설명 |
|------|--------|------|
| **프로젝트 시작** (신규/리뉴얼) | Track A | 전체 범위를 정의하고 Jira 구조를 일괄 생성한 후 순차 개발 |
| **기능 추가/버그 수정** (운영 중) | Track B | 개별 아이디어에서 Jira 단건 생성 후 개발 |

## Track A: 프로젝트 시작 (신규/리뉴얼)

신규 프로젝트 또는 레거시 리뉴얼 시의 흐름.
전체 서비스를 한번에 분석하고 구조화한 후 순차적으로 개발한다.

```
프로젝트 정의 (사람)
    ↓
AI 레거시 분석 / 전체 범위 정의
    ↓
AI 기술 스택 결정 + 시스템 설계
    ↓
사람 승인 (CP0: 프로젝트 방향 승인)
    ↓
AI 기능 분해 → Epic / Story / Task 일괄 생성
    ↓
Jira 구조 일괄 생성 + 우선순위 정렬
    ↓
사람 확인 (CP1: 전체 범위 확인)
    ↓
디자인 시안 — 핵심 화면 비주얼 방향 확정 (사람 승인)
    ↓
Sprint 단위로 Track B의 개발 흐름 반복 실행
```

### A-1. 프로젝트 정의

사람이 프로젝트의 방향을 전달한다.

입력 형태는 자유롭다:
- "기존 MocaKey 서비스를 새로 만들 거야"
- "이 레거시 코드를 리뉴얼하고 싶어" + 코드 경로
- "이런 서비스를 처음부터 만들고 싶어" + 기획 문서

AI가 처리 가능한 최소 입력:
- **무엇을 만들 것인가** (1문장 이상)
- 레거시 리뉴얼 시: 기존 코드 경로

### A-2. AI 전체 분석

AI가 프로젝트 전체를 분석한다. (05_ai_planning_phase 참조)

- 레거시 리뉴얼 시: 기존 코드 규모 측정, 기술 스택 식별, 핵심 기능 추출, 외부 연동 식별
- 신규 시: 요구사항 구조화, 도메인 분석, 유사 시스템 패턴 조사

### A-3. 기술 스택 + 시스템 설계

AI가 자율적으로 판단한다. (05_ai_planning_phase의 기술 스택 선정 기준 적용)

- 기술 스택 선정 (프레임워크, DB, 언어)
- 아키텍처 패턴 결정 (모놀리스/마이크로서비스)
- 프로젝트 구조 설계 (모노레포/멀티레포)
- SDK/외부 연동 인터페이스 방침
- 디자인 시스템 기본 규격

### A-4. CP0: 프로젝트 방향 승인

사람이 전체 방향을 확인하고 승인한다.

- 확인 대상: 기술 스택, 아키텍처, 리뉴얼 범위, 실행 일정
- **프로젝트에서 가장 중요한 의사결정 포인트** — 방향이 틀리면 전체가 틀어진다
- 거부 시: 사람이 방향을 수정하고 AI가 재분석

### A-5. 기능 분해 + Jira 일괄 생성

AI가 전체 기능을 Jira 구조로 분해한다.

```
전체 서비스
    ↓
Epic (대분류): 인증, 사용자 관리, 카드 관리, 디바이스 관리, ...
    ↓
Story (기능 단위): 로그인, 회원가입, 카드 발급, 카드 조회, ...
    ↓
Task (작업 단위): [backend] API 구현, [ios] 화면 구현, [web] 페이지 구현, ...
```

- 각 티켓은 06_jira_template의 필수 항목 + 메타 필드를 자동으로 채운다
- Epic → Story → Task 의존성을 자동 설정한다
- 플랫폼별 태스크를 분리한다 (backend 먼저, 앱/웹은 API 의존)
- 우선순위를 자동 정렬한다 (11_priority_rules 참조)

### A-6. CP1: 전체 범위 확인

사람이 생성된 Jira 구조를 확인한다.

- 기능 누락 여부
- 우선순위 적절성
- 범위 과잉/부족

확인 후 디자인 시안을 먼저 확정하고, Sprint 단위로 개발을 시작한다.

### A-7. 디자인 시안 — 비주얼 방향 확정

신규 프로젝트에서 개별 기능을 구현하기 전에, 핵심 화면의 비주얼 방향을 먼저 확정한다.
홈 화면 등 대표 화면의 디자인이 정해지면, 나머지 화면은 같은 톤으로 진행할 수 있어 개별 티켓마다 비주얼 검토를 반복할 필요가 줄어든다.

```
CP1 (전체 범위 확인) 완료
    ↓
Designer AI가 핵심 화면 디자인 시안 생성
    - 홈 화면 (앱의 전체 톤 확정)
    - 주요 플로우 1~2개 (리스트, 상세, 입력 폼 등)
    - 디자인 토큰 (색상, 타이포, 간격, 라운딩 등)
    ↓
사람이 비주얼 방향을 확인하고 승인
    ↓
승인된 시안이 디자인 시스템의 기준이 됨
    ↓
Sprint 시작 — 개별 티켓은 이 기준을 따라 구현
```

#### 산출물

- `artifacts/00_init/design_concept.md` — 비주얼 방향 정의 (컬러, 타이포, 컴포넌트 스타일)
- `artifacts/00_init/key_screens/` — 핵심 화면 시안 (코드 또는 이미지)
- `docs/design-tokens.md` — 디자인 토큰 정의 (Designer AI와 Dev AI가 참조)

#### 개별 티켓 디자인과의 관계

디자인 시안이 확정된 후:
- **기존 패턴 내 화면**: Designer AI가 시안 기준으로 바로 구현. 별도 비주얼 검토 불필요
- **새로운 UX 패턴 도입**: 와이어프레임 + 사람 확인 후 구현
- Track B(운영 중 기능 추가)에서도 이 디자인 시스템을 기준으로 한다

### A-8. 백로그 변동 관리

일괄 생성된 Jira 구조는 Sprint 진행 중 변동이 발생할 수 있다. 백로그와 현실의 괴리를 방지하기 위해 다음 규칙을 적용한다.

#### 변동 유형별 처리

| 변동 유형 | 처리 방식 | 승인 |
|----------|----------|:----:|
| 새 기능 추가 (Story/Task) | AI가 기존 Epic 구조 내 적절한 위치를 제안 → 사람 확인 후 추가 | 사람 |
| 요구사항 변경 (기존 티켓 수정) | AI가 영향 범위 분석 후 관련 티켓도 함께 업데이트 제안 | 사람 |
| 우선순위 재정렬 | 사람이 직접 변경하거나 AI에게 재정렬 요청 | 사람 |
| 티켓 삭제/보류 | 사람만 가능. AI는 삭제를 제안할 수 있으나 실행하지 않음 | 사람 |
| 티켓 분할 (너무 큰 작업) | AI가 분할 안을 제안 → 사람 확인 후 실행 | 사람 |

#### Sprint 시작 시 점검

각 Sprint 시작 전 다음을 확인한다:

- 해당 Sprint에 포함된 티켓이 여전히 유효한가
- 의존성이 해소되었는가 (선행 작업 완료 여부)
- 우선순위가 현재 비즈니스 상황과 맞는가

#### 원칙

- AI는 백로그를 **제안**할 수 있으나, 구조 변경(추가/삭제/이동)은 **사람이 승인**한다
- 변경 이력은 Jira 코멘트로 남겨 추적 가능하게 한다
- Epic/Story 수준의 구조 변경은 CP1을 다시 거친다

## Track B: 기능 개발 (공통 개발 흐름)

Track A에서 생성된 Jira 티켓이든, 운영 중 추가된 아이디어든, 개별 기능 개발은 이 흐름을 따른다.

### 운영 중 기능 추가 시 진입

```
아이디어 (사람)
    ↓
AI 기획 보조 → 기획서 초안 → 사람 검토 (05_ai_planning_phase 참조)
    ↓
Jira 티켓 자동 생성 (06_jira_template 참조)
    ↓
아래 공통 개발 흐름으로 진입
```

### 공통 개발 흐름

```
Jira 티켓
    ↓
복잡도 판별 (단순/보통/복잡)
    ↓
AI Jira 확장 → CP1: 요구사항 확인 (단순: Ideator+Planner 통합)
    ↓
계획 수립 → CP2: 계획 승인 (risk_level별)
    ↓
교차 리뷰 → 프로토타입 필요성 자동 판정
    ↓
프로토타입 (리스크 기반 자동 판정)
    ├── 기술 리스크 → 기술 검증 PoC (미확인 항목 최소 코드 검증)
    ├── UX 리스크 → 디자인 시안 (HTML, 2~3개 구조 다른 안)
    ├── 둘 다 → PoC + 디자인 시안
    └── 없음 → 스킵
    ↓
1차 디자인 — 와이어프레임 (조건부)
    ↓
테스트 설계 → Test Validator 자동 검증 → 테스트 리뷰 (CP2.5: 테스트 검증)
    ↓
구현 (테스트 통과 목표)
    ↓
리뷰 → CP3: 구현 리뷰 (가드레일/고위험)

※ risk_level에 따라 단계를 경량화한다:
  low:      테스트 → 구현 → 리뷰(통합 1회)  ~150K/story
  medium:   테스트 + Validator → 구현 → 리뷰(통합 1회)  ~400K/story
  high:     풀 프로세스 (Validator + 교차 리뷰 + 리뷰 1차+2차)  ~800K/story
    ↓
CI/CD → 실패 시 자동 수정 / 에스컬레이션
    ↓
CP4: 배포 승인 (플랫폼별 차등)
    ↓
안정화 → CP5: 안정화 확인
```

#### 2. 계획 수립

#### 2.5. 프로토타입 (조건부)

requires_ui_change == yes이고, 새로운 UX 패턴이나 동작 확인이 필요한 경우 AI가 빌드 가능한 프로토타입을 먼저 만든다.
문서만으로는 "이게 맞는지" 판단이 어렵기 때문에, 실제 기기/시뮬레이터에서 돌려보고 방향을 확정한다.

```
Spec AI 설계 완료
    ↓
AI가 프로토타입 대상인지 판정
    ↓
AI가 프로토타입 코드 작성 (별도 브랜치: feature/{ticket}-prototype)
  - 화면 구현 + 핵심 기능만 최소화하여 포함
  - 더미 데이터 사용, 에러 처리/테스트/품질 관리 생략
    ↓
개발자가 빌드 → 시뮬레이터/기기에서 실행하여 확인
    ↓
사람 확인: "이 방향이 맞는가?"
    ↓
승인 → 프로토 브랜치 삭제, 본 브랜치에서 정식 구현 시작
반려 → 피드백 반영하여 재작업 (최대 3회)
```

##### 프로토타입 정의

**프로토타입 = 화면 구현 + 핵심 기능 최소화**

빌드해서 직접 돌려볼 수 있는 수준의 코드를 만든다. 프로토타입은 검증 도구이지 최종 코드가 아니다.

| 포함하는 것 | 포함하지 않는 것 |
|------------|----------------|
| UI 레이아웃 + 핵심 인터랙션 | 에러 처리, 엣지 케이스 |
| 화면 전환 / 네비게이션 | 테스트 코드 |
| 핵심 기능 1개의 최소 동작 | 전체 기능, 완성도 높은 구현 |
| 더미 데이터로 화면 표시 | 실제 API/DB 연동 (불필요 시) |
| 빌드 가능한 상태 | 코드 리뷰 수준의 품질 |

##### 프로토타입 대상 판정

| 조건 | 프로토타입 필요 | 예시 |
|------|:--------------:|------|
| 새로운 UI 컴포넌트/패턴 | O | 위젯, 커스텀 인터랙션, 새 화면 유형 |
| 사용자 플로우가 복잡 | O | 멀티 스텝 폼, 결제 플로우, 온보딩 |
| 기존 패턴 내 단순 화면 | X | CRUD 목록/상세, 설정 토글 |
| UI 변경 없는 백엔드 작업 | X | API 추가, 비즈니스 로직 수정 |

AI가 자동 판정하고, 사람이 CP2에서 재지정할 수 있다.

##### 플랫폼별 실행 방법

| 플랫폼 | 프로토타입 방법 | 확인 방법 |
|--------|---------------|----------|
| **iOS** | 타겟/화면 추가 + 더미 데이터 | Xcode → Run → 시뮬레이터/기기에서 확인 |
| **Android** | Activity/Fragment 추가 + 더미 데이터 | Android Studio → Run → 에뮬레이터/기기에서 확인 |
| **Web** | 페이지 추가 + 더미 데이터 | `pnpm dev` → 브라우저에서 확인 |
| **Backend** | API 엔드포인트 + Mock 응답 | `curl` 또는 Swagger UI에서 확인 |

##### 프로토타입 후 코드 처리

프로토타입 코드는 **버린다**. 확정된 방향을 기반으로, 본 브랜치에서 정식 프로세스(테스트 선행 → 구현 → 리뷰)로 처음부터 구현한다.

```
프로토타입 승인
    ↓
feature/{ticket}-prototype 브랜치 삭제
    ↓
feature/{ticket} 브랜치에서 정식 개발 시작
    ↓
Designer → Test → Dev → Review (정식 프로세스)
```

##### 프로토타입 산출물

- `artifacts/{ticket_id}/02.5_prototype/prototype_summary.md` — 프로토타입 설명, 빌드/실행 방법, 확인 포인트
- 프로토타입 코드 (별도 브랜치 `feature/{ticket}-prototype`)

##### 사람 확인 포인트

- "이 화면/동작이 원하는 방향인가?"
- "빠진 기능이나 잘못된 흐름이 있는가?"
- 확인 후 피드백을 주면 AI가 반영하여 재작업

#### 3. 1차 디자인 — 와이어프레임 (조건부)

requires_ui_change == yes인 경우 Designer AI가 UI/UX 플로우와 와이어프레임을 생성한다.
구조와 플로우를 확정하여 구현 방향을 잡는다.
새로운 UX 도입 시 사람 확인이 필요하다.
프로토타입이 선행된 경우, 프로토타입에서 확정된 방향을 기반으로 와이어프레임을 작성한다.

#### 4. 테스트 설계

AI가 명세(설계 문서, 인수 조건)를 기반으로 테스트 케이스와 테스트 코드를 먼저 작성한다.
테스트는 기능 구현 전에 완성되어야 하며, 빈 구현체에 대해 실패하는 상태여야 한다.

##### 테스트 완전성 기준

**"단위 테스트만 있으면 충분하다"는 판단을 하지 않는다.** 다음 기준을 모두 충족해야 테스트 설계가 완료된 것이다.

| 기준 | 설명 |
|------|------|
| **인수 조건 전수 매핑** | 모든 인수 조건에 1:1 대응하는 테스트가 있어야 한다. 매핑표를 산출물 첫 번째 섹션으로 작성한다 |
| **5개 카테고리 검토** | 단위 / 통합·트리거 / UI 상태 / 생명주기 / 에러·경계값 — 5개 카테고리를 전부 검토하고, 해당하는 테스트를 작성한다 |
| **수정 검증 가능** | AI가 코드를 수정했을 때, 수정이 반영되었는지 테스트로 직접 확인할 수 있어야 한다. View 인스턴스만 생성하고 내용을 검증하지 않는 테스트는 불완전하다 |
| **결과값 assert** | 가능한 한 구체적인 값(텍스트, visibility, 호출 횟수 등)을 assert한다. "크래시 없음"만으로는 부족하다 |
| **플랫폼 일관성** | iOS/Android 동일 기능은 동일한 시나리오를 검증한다 |

##### 테스트 카테고리 정의

| 카테고리 | 검증 대상 | 예시 |
|---------|----------|------|
| **단위 테스트** | 순수 함수, 데이터 변환, 비즈니스 로직 | `countActiveCards()`, 데이터 저장/조회 |
| **통합·트리거 테스트** | 이벤트 발생 시 연쇄 동작이 실행되는지 | 카드 추가 → 위젯 갱신 호출, 앱 포그라운드 → 데이터 동기화 |
| **UI 상태 테스트** | 데이터 상태 → UI 렌더링 결과 | 미로그인 → "로그인 필요" 텍스트 + 버튼 숨김, RemoteViews visibility |
| **생명주기 테스트** | 컴포넌트 시작/종료/타임아웃/콜백 | Service 시작 → Foreground 전환, 10초 타임아웃 → 정리, guard 조건 |
| **에러·경계값 테스트** | 비정상 입력, 실패 시나리오, 경계 조건 | 빈 데이터, null, 권한 없음, BLE 실패, 이중 호출 방지, 0건/1건/N건 |

#### 4.5. 테스트 리뷰 (CP2.5)

테스트 코드가 명세를 정확히 반영하는지 리뷰한다.
테스트가 잘못되면 구현도 잘못된 방향으로 진행되므로, 구현에 앞서 테스트의 정확성을 검증한다.

**먼저 Test Validator가 구조적 품질을 자동 검증**한 후, 교차 리뷰로 넘어간다.

```
Test AI 테스트 코드 작성 완료
    ↓
[자동 검증] Test Validator AI — 구조적 품질 사전 검증
    ├── V-1~V-4: 구조 (AC 매핑, 카테고리, Traceability)
    ├── V-5~V-7: 코드 품질 (Placeholder, Assert, 동작 검증)
    ├── V-8~V-10: API 정합성 (Mock 일치, 상수, import)
    └── V-11~V-12: 플랫폼 일관성
    ↓
    반려 → Test AI 재실행 (Validator 피드백 전달)
    통과 ↓
[1차] 1차 리뷰어 — 테스트 설계 검증
    ├── 인수 조건별 테스트 존재 여부
    ├── 엣지 케이스 / 에러 케이스 누락
    ├── 플랫폼 간 테스트 시나리오 일관성 (iOS/Android 동일 검증)
    └── 테스트가 검증하는 동작이 명세와 일치하는지 (추측 기반 아닌지)
    ↓
[2차] 2차 리뷰어 — 테스트 코드 품질 검증
    ├── 테스트 코드 자체의 버그
    ├── 모킹/스텁 설정이 실제 동작과 괴리가 없는지
    ├── 비결정적 테스트 (flaky test) 위험
    └── 테스트 성능 (불필요하게 느린 테스트)
    ↓
종합 판정 → 승인 시 구현 착수
    ↓
반려 시 → Test AI가 수정 → 재리뷰 (최대 3회, 이후 에스컬레이션)
```

##### 테스트 리뷰 체크리스트

| 항목 | 확인 내용 |
|------|----------|
| **인수 조건 매핑표** | 산출물 첫 번째 섹션에 인수 조건 ↔ 테스트 매핑표가 있는가. **없으면 즉시 반려** |
| **매핑 누락** | 매핑표에 "매핑 안 된 인수 조건: 없음"이 명시되어 있는가. 누락된 인수 조건이 있으면 반려 |
| **5개 카테고리 검토** | 단위 / 통합·트리거 / UI 상태 / 생명주기 / 에러·경계값 — 5개 카테고리를 전부 검토했는가 |
| **수정 검증 테스트** | AI가 코드 수정 후 변경 반영을 확인할 수 있는 테스트가 있는가 |
| **결과값 assert** | View 생성만 확인하고 내용을 검증하지 않는 테스트가 있으면 지적 |
| 정상 동작 | 성공 시나리오를 테스트하는가 |
| 에러 / 예외 | 실패 시나리오, 유효하지 않은 입력을 테스트하는가 |
| 경계값 | 빈 값, 최대값, 0건, 1건, N건 등 경계를 테스트하는가 |
| 플랫폼 일관성 | iOS/Android 동일 기능의 테스트가 동일한 시나리오를 검증하는가 |
| 명세 정합성 | 테스트가 검증하는 동작이 명세(설계 문서)와 정확히 일치하는가 |
| 독립성 | 테스트 간 순서 의존 없이 독립 실행 가능한가 |
| 기존 테스트 | 기존 테스트가 깨지지 않는가 |
| **테스트 실행 확인** | 테스트를 실제로 실행하여 전체 통과했는가. **실행 0개 = 즉시 반려**. 빌드 성공 ≠ 테스트 통과 |
| **프로젝트 등록** | 테스트 파일이 빌드 시스템에 등록되어 있는가 (iOS: `.pbxproj` 포함, Android: `src/test/` 경로) |
| **import 경로** | `@testable import`/모듈 참조가 테스트 타겟에서 접근 가능한가 (iOS: Widget Extension 모듈은 앱 호스트에서 import 불가) |
| **자기 검증** | Test AI의 자기 검증 체크리스트가 전부 체크되어 있는가 |

##### 테스트 리뷰 산출물

- artifacts/{ticket_id}/04_test/primary_test_review.md (1차)
- artifacts/{ticket_id}/04_test/secondary_test_review.md (2차)
- 판정: 승인 / 반려 (반려 시 수정 사항 명시)

#### 5. 구현

테스트 리뷰를 통과한 테스트 코드를 기준으로 AI가 기능 코드를 구현한다.
모든 테스트가 통과하는 것을 구현 완료 기준으로 한다.

##### 결정 근거 문서화 (Decision Record)

AI가 설계나 구현 시 **왜 이 방식을 선택했는지** 기록한다. 리뷰에서 결정의 타당성과 대안 여부를 검증하기 위함이다.

각 단계에서 다음을 산출물에 포함한다:

| 단계 | 기록 위치 | 기록 내용 |
|------|----------|----------|
| Spec AI (설계) | design_doc.md 내 "결정 근거" 섹션 | 아키텍처 선택, API 설계 방식, DB 스키마 설계 이유 |
| Dev AI (구현) | change_summary.md 내 "결정 근거" 섹션 | 구현 패턴, 라이브러리 선택, 알고리즘 선택 이유 |

기록 형식:

```
### 결정 근거

| 결정 사항 | 선택한 방식 | 이유 | 검토한 대안 | 대안을 선택하지 않은 이유 |
|----------|-----------|------|-----------|----------------------|
| 캐싱 방식 | Redis | 분산 환경 지원, TTL 관리 용이 | 인메모리 캐시 | 서버 재시작 시 유실, 다중 인스턴스 불일치 |
| 인증 방식 | JWT | 무상태, 마이크로서비스 확장성 | 세션 기반 | 서버 부하, 수평 확장 시 세션 공유 문제 |
```

기록 기준:
- **항상 기록**: 라이브러리/프레임워크 선택, 아키텍처 패턴, DB 스키마 설계, API 설계 방식
- **선택적 기록**: 단순 구현 (패턴이 명확하고 대안이 없는 경우)
- 대안이 명확히 없는 단순 구현은 기록하지 않아도 된다

#### 6. 구현 리뷰 (CP3)

구현된 코드가 명세를 충족하고 품질 기준을 통과하는지 검증한다.
**1차 리뷰어 + 2차 리뷰어 교차 리뷰**로 단일 모델의 맹점을 보완한다.
**CP3는 AI가 전부 리뷰한다.** 사람에게는 AI가 "꼭 확인이 필요하다"고 판단한 항목만 전달한다.

```
Dev AI 구현 완료 (모든 테스트 통과)
    ↓
[1차] 1차 리뷰어 — 요구사항·구조·플랫폼 검증
    ├── 인수 조건 충족 여부
    ├── 코드 구조 / 설계 패턴
    ├── iOS/Android 동일 구현 여부 (플랫폼 일관성)
    ├── 가드레일 영역 변경 감지
    ├── 테스트 커버리지 확인
    └── 플랫폼별 필수 규칙 준수 (아래 참조)
    ↓
[2차] 2차 리뷰어 — 보안·성능·엣지케이스 검증
    ├── 보안 취약점
    ├── 성능 문제
    ├── 엣지 케이스 누락
    └── 코드 스타일 / 컨벤션
    ↓
[종합 판정] AI가 최종 결정
    ├── 둘 다 승인 → 통과 (사람 확인 불필요)
    ├── 하나라도 반려 → 반려 사유 통합하여 Dev AI에 전달
    └── "사람 확인 필요" 항목이 있으면 → 해당 항목만 사람에게 전달
    ↓
반려 시 → Dev AI가 수정 → 변경 부분만 재리뷰 (최대 3회, 이후 에스컬레이션)
```

##### 1차 리뷰 체크리스트

| 카테고리 | 확인 항목 |
|---------|----------|
| **요구사항** | 모든 인수 조건이 구현되었는가 |
| **요구사항** | 명세에 없는 동작을 임의로 추가하지 않았는가 |
| **구조** | 기존 아키텍처 패턴을 따르는가 |
| **결정 근거** | 설계/구현 결정에 대한 근거가 기록되어 있는가 |
| **대안 검토** | 다른 합리적인 방식이 있는데 검토 없이 넘어간 것은 없는가 |
| **가드레일** | DB 스키마, 인증, 결제, 보안, 인프라 영역에 변경이 있는가 |
| **테스트** | 테스트 커버리지가 기준(80%)을 충족하는가 |
| **테스트 실행 결과** | change_summary에 "테스트 N개 실행, N개 통과"가 명시되어 있는가. **빌드 성공만 기재하고 테스트 실행 결과가 없으면 반려** |
| **수정 검증** | change_summary의 "수정 ↔ 테스트 검증 상태"를 확인. 테스트 미커버 항목이 있으면 위험도를 판단한다 |
| **플랫폼 일관성** | iOS/Android 동일 기능이 동일하게 구현되었는가 |
| **플랫폼 교차 실행** | iOS/Android 양쪽 모두에서 테스트를 실행하여 통과했는가. 한쪽만 실행한 경우 반려 |
| **플랫폼 규칙** | 아래 "플랫폼별 필수 규칙"을 준수하는가 |
| **기존 동작** | 기존 기능에 사이드 이펙트가 없는가 |

##### 2차 리뷰 체크리스트

| 카테고리 | 확인 항목 |
|---------|----------|
| **보안** | SQL 인젝션, XSS, 인증 우회 등 취약점이 있는가 |
| **보안** | 민감 정보가 로그나 응답에 노출되는가 |
| **성능** | N+1 쿼리, 불필요한 반복, 메모리 누수가 있는가 |
| **엣지 케이스** | null/undefined, 빈 배열, 동시성 등 처리가 되어 있는가 |
| **코드 스타일** | 프로젝트 컨벤션(네이밍, 포맷, 린트)을 따르는가 |

##### 플랫폼별 필수 규칙

AI가 리뷰 시 자동으로 검증하는 플랫폼 고유 규칙이다.

**iOS**

| 카테고리 | 규칙 | 안 하면 발생하는 문제 |
|---------|------|---------------------|
| 프로젝트 파일 | 새 소스 파일은 `.pbxproj`의 4곳(FileRef, BuildFile, Group, Sources phase)에 등록 | 파일이 디스크에 있어도 빌드에 포함 안 됨. 테스트 0개 실행 |
| Extension 타겟 | Widget/Notification Extension은 별도 프로세스. 공유 코드는 양쪽 타겟에 포함 | `@testable import ExtensionModule`이 앱 호스트 테스트에서 런타임 로드 실패 |
| 시뮬레이터 | `#if targetEnvironment(simulator)` 사용. `arch(x86_64)` 분기는 Apple Silicon에서 작동 안 함 | 탈옥 감지 등이 시뮬레이터를 탈옥으로 오판 → `exit(0)` → 테스트 실행 불가 |
| 권한 | 새 권한 사용 시 `Info.plist`에 Usage Description 추가 | 앱스토어 리젝트 |
| Capability | App Groups, Keychain 등은 앱 + Extension 양쪽 Entitlements 동일 설정 | 데이터 공유 실패 |
| 메모리 | 클로저 캡처 시 `[weak self]` (retain cycle 방지). `static`/`struct`는 불필요 | 메모리 누수 |
| 메인 스레드 | UI 업데이트는 `@MainActor` 또는 `DispatchQueue.main` | UI 업데이트 무시 또는 크래시 |
| Swift 호환 | non-optional 값에 `if let`/`?.` 사용 금지. 최신 Swift에서 컴파일 에러 | 빌드 실패 |
| 접근성 | UI 요소에 `accessibilityLabel` 설정 | 접근성 감사 실패 |

**Android**

| 카테고리 | 규칙 | 안 하면 발생하는 문제 |
|---------|------|---------------------|
| Manifest | 새 Activity/Service/Receiver는 `AndroidManifest.xml` 등록. ForegroundService는 `foregroundServiceType` 필수 (API 29+) | 컴포넌트 미발견 크래시 |
| 권한 | `<uses-permission>` + Runtime permission 요청. `FOREGROUND_SERVICE_*` 세부 권한은 API 34+ 필수 | SecurityException 크래시 |
| 테스트 경로 | 단위: `src/test/` (JVM), 통합: `src/androidTest/` (에뮬레이터). BuildVariant별 sourceSet 주의 | 테스트 미발견, 0개 실행 |
| Context | `Context` 필요 테스트는 Robolectric 설정 필수. 싱글턴에 `Activity` Context 저장 금지 | 테스트 크래시, 메모리 누수 |
| ProGuard | 새 라이브러리 추가 시 R8 규칙 확인 | Release 빌드에서 클래스 못 찾음 |
| 메인 스레드 | 네트워크/DB는 `Dispatchers.IO` 코루틴 | ANR (Application Not Responding) |
| PendingIntent | `FLAG_IMMUTABLE` 또는 `FLAG_MUTABLE` 필수 (API 31+) | 크래시 |
| ForegroundService | `startForegroundService()` 후 5초 내 `startForeground()` 호출 | ANR + 시스템 강제 종료 |
| 접근성 | UI 요소에 `contentDescription` 설정 | 접근성 감사 실패 |

**백엔드**

| 규칙 | 내용 |
|------|------|
| API 스펙 | 새 엔드포인트 추가 시 OpenAPI 스펙 업데이트 |
| 환경 변수 | 새 환경 변수 추가 시 `.env.example` 업데이트 |
| 마이그레이션 | DB 스키마 변경 시 마이그레이션 파일 생성 (직접 DB 수정 금지) |
| 에러 코드 | 새 에러 상황은 표준 에러 코드 체계에 등록 |
| 로깅 | 새 API에 요청/응답 로깅 설정 |

**웹**

| 규칙 | 내용 |
|------|------|
| 라우팅 | 새 페이지 추가 시 라우터에 등록 |
| 환경 변수 | 클라이언트 노출 변수는 `NEXT_PUBLIC_` 접두사 |
| SEO | 새 페이지에 `<title>`, `<meta description>` 설정 |
| 접근성 | 인터랙티브 요소에 ARIA 속성, 키보드 네비게이션 |
| 반응형 | 모바일/태블릿/데스크톱 브레이크포인트 확인 |

##### "사람 확인 필요" 기준

AI가 리뷰를 전부 수행한 후, 다음에 해당하는 항목만 사람에게 전달한다.
**해당 항목이 없으면 사람 확인 없이 통과한다.**

```
사람에게 전달하는 것:
1. 가드레일 영역 변경 (DB, 인증, 결제, 보안, 인프라)
2. AI가 판단하기 어려운 비즈니스 로직 정확성
3. iOS/Android 간 동작 차이가 감지되었으나 의도 여부를 모를 때
4. 아키텍처에 큰 영향을 미치는 설계 변경

사람에게 전달하지 않는 것:
- 코드 스타일 (린터가 처리)
- 테스트 커버리지 (AI가 측정)
- 정적 분석 (도구가 처리)
- 플랫폼 규칙 위반 (AI가 직접 반려)
- 보안/성능 문제 (AI가 직접 반려)
```

##### 반려 → 재작업 흐름

```
리뷰 반려
    ↓
반려 사유를 Dev AI에 전달 (구체적 수정 방향 포함)
    ↓
Dev AI 코드 수정 → 변경 부분만 재리뷰
    ↓
3회 반려 시 → 에스컬레이션 (09_escalation_policy)
```

##### 구현 리뷰 산출물

- artifacts/{ticket_id}/07_review/primary_review.md (1차)
- artifacts/{ticket_id}/07_review/secondary_review.md (2차)
- artifacts/{ticket_id}/07_review/review_summary.md (종합 — 승인/반려/사람 확인 필요 항목 포함)

#### 7. CI/CD

자동화된 테스트 파이프라인이 실행된다. (16_ai_dev_ci_cd 참조)

#### 8. 배포

승인된 변경 사항이 스테이징 또는 프로덕션에 배포된다.
플랫폼별 배포 승인 방식이 다르다. (08_human_checkpoint 참조)

#### 9. 안정화

프로덕션 배포 후 안정화 구간을 운영한다. (18_post_launch 참조)
모니터링, 핫픽스, 사용자 피드백 수집을 수행하고, 안정화 기준 충족 시 종료한다.
