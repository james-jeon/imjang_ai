# AI 개발 CI/CD 연동

## 공통 파이프라인

모든 플랫폼에 공통으로 적용되는 단계:

```
PR 생성
  ↓
Lint
  ↓
단위 테스트
  ↓
통합 테스트
  ↓
보안 스캔
  ↓
빌드
  ↓
스테이징 배포
```

## 플랫폼별 파이프라인

### 백엔드

```
PR 생성
  ↓
Lint + 정적 분석
  ↓
단위 테스트
  ↓
통합 테스트 (DB, 외부 API 연동)
  ↓
API 계약 테스트
  ↓
보안 스캔 (의존성 취약점 + SAST)
  ↓
부하 테스트 (주요 변경 시)
  ↓
빌드 + Docker 이미지 생성
  ↓
스테이징 배포
  ↓
스모크 테스트
  ↓
모니터링/알림 설정 확인
```

### 앱 (iOS / Android)

```
PR 생성
  ↓
Lint + 정적 분석
  ↓
단위 테스트
  ↓
UI 테스트
  ↓
보안 스캔
  ↓
플랫폼별 빌드 (iOS + Android 병렬)
  ↓
UI 스냅샷 테스트 (주요 해상도별)
  ↓
디바이스 호환성 테스트 (테스트 매트릭스)
  ↓
스테이징 배포 (TestFlight / Internal Track)
  ↓
스토어 심사 제출 준비 (Release AI 산출물 확인)
```

#### 앱 전용 고려사항
- iOS/Android 빌드는 병렬로 실행한다
- 인증서/프로비저닝 프로파일 만료 여부를 빌드 전 자동 확인한다
- 스토어 심사 제출은 사람 승인 후 실행한다
- 단계적 출시 (Phased Release)를 지원한다

#### 앱 CI/CD 구현 현황

> 워크플로우 파일, fastlane 명령, 시크릿 등 구현 상세는 **14_tool_integration.md**의 "앱 CI/CD 구현 현황" 참조

## 프로덕션 배포 후 안정화 모니터링 활성화

프로덕션 배포가 완료되면 CI/CD 파이프라인이 다음을 자동으로 트리거한다:

- Release AI가 모니터링 대시보드/알림을 설정한다 (18_post_launch 참조)
- 안정화 모니터링 구간이 시작된다 (에러율, 크래시율, 응답 시간 추적)
- 이상 징후 감지 시 핫픽스 프로세스가 자동 진입한다
- 안정화 완료 기준 충족 시 CP5로 사람에게 확인을 요청한다 (08_human_checkpoint 참조)

---

## AI의 역할

- AI가 플랫폼별 파이프라인 실행을 트리거한다
- 멀티 플랫폼 작업 시 백엔드 → 앱/웹 순서로 배포를 조율한다
- 실패 로그를 분석하여 문제를 자동으로 수정한다
- 프로덕션 배포 완료 후 안정화 모니터링을 활성화한다

## 실패 처리 정책

### 자동 수정 (AI)

| 실패 단계 | AI 조치 |
|-----------|---------|
| Lint | 코드 스타일 자동 수정 후 재실행 |
| 단위 테스트 | 실패 원인 분석 → 코드 수정 → 재실행 |
| 통합 테스트 | 실패 원인 분석 → 코드 수정 → 재실행 |
| UI/E2E 테스트 | 스냅샷 비교 → UI 코드 수정 → 재실행 |
| 크로스브라우저 | 브라우저별 호환성 코드 수정 → 재실행 |
| 보안 스캔 | 취약점 유형 확인 → 코드 수정 → 재실행 |
| 빌드 | 빌드 오류 분석 → 코드 수정 → 재실행 |
| 접근성 테스트 | 접근성 위반 항목 수정 → 재실행 |
| 성능 테스트 | 성능 병목 분석 → 최적화 → 재실행 |

### 재시도 제한

- 각 단계별 최대 재시도: 2회
- 2회 재시도 후에도 실패 시 에스컬레이션 (09_escalation_policy 참조)

### 롤백 정책

- 스테이징 배포 실패 시: 자동 롤백 후 AI에게 실패 로그 전달
- 프로덕션 배포 실패 시: 즉시 자동 롤백 + 사람에게 알림
- 앱 스토어 심사 거절 시: 거절 사유 분석 → AI가 수정안 제시 → 사람 확인 후 재제출

### 사람 개입 시점

- 보안 스캔에서 critical 취약점 발견 시: 자동 수정하지 않고 사람에게 보고
- 2회 연속 실패 후: 에스컬레이션
- 프로덕션 배포 실패 후: 사람이 원인 확인 후 재배포 판단
- 앱 스토어 제출: 사람 승인 필수
- 부하 테스트 기준 미달: 사람에게 보고 후 판단

### 실패 보고 형식

AI가 실패를 보고할 때는 다음 구조를 따른다. (상세 출력 형식은 13_output_format 참조)

```
[파이프라인 실패 보고]

플랫폼: backend / app-ios / app-android / web
실패 단계: 단위 테스트
실패 요약: UserServiceTest.testCancelReservation - NPE 발생
원인 분석: cancelReservation() 내 status 필드가 null인 케이스 미처리
시도한 수정: status null 체크 추가 (1차), 기본값 설정 (2차)
현재 상태: 2회 재시도 실패 → 에스컬레이션
권장 조치: ReservationStatus enum에 UNKNOWN 기본값 추가 검토 필요
```
